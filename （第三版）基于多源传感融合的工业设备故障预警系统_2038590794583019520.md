# 摘要

针对工业泵房场景中水泵驱动电机及其配电系统故障定位困难、人工排查效率低、非计划停机影响供水连续性等问题，本文面向“工业控制智能应用”赛题中的设备故障预警方向，提出一种基于多源传感融合的泵房设备故障预警系统方案。该方案以振动、温度和电流等传感器数据为核心输入，结合泵房典型故障机理，构建“传感采集—数据处理—故障分类—趋势预测—分级预警—控制联动—HMI 显示”的整体架构。算法层采用 SVM 作为基线分类模型、随机森林作为主分类模型、LSTM 作为趋势预测模型，以实现对电机绕组开路、绕组不平衡与匝间短路、水泵堵转及连接机构断开等异常状态的识别与提前预警；系统层采用软件侧与控制侧分层设计，软件侧完成数据分析与模型推理，并通过 MQTT 与 EAE 平台交换数据，控制侧负责 PLC 逻辑、I/O 执行与HMI 显示。本文进一步给出了数据预处理、特征提取、通信机制、控制响应逻辑及MATLAB/Simulink 仿真验证路线。研究结果表明，该方案具有较好的技术可行性、工程扩展性和应用推广价值，可为泵房及类似旋转类机电设备的预测性维护和智能运维提供参考。

关键词：工业控制；设备故障预警；泵房设备；多源传感融合；随机森林；LSTM；MQTT；EAE

# 第一章 项目概述

# 1.1 选题背景

随着工业自动化和智能制造的持续发展，以电机为核心驱动的旋转设备（如水泵、风机和输送机）在制造业、能源系统以及污水处理等领域中发挥着关键作用。这类设备通常承担连续输送和工艺执行任务，其运行状态直接影响生产效率、工艺稳定性以及整体系统的安全性。

在实际运行过程中，这些设备往往处于长时间连续工作、负载波动明显以及环境复杂的工况之下。诸如轴承磨损、转子不平衡、机械卡滞或过载发热等故障，通常并不会突然发生，而是会先通过一些较为隐蔽的早期信号表现出来，例如振动异常、温度上升或电流波动等。如果缺乏有效的监测与预警手段，这些“弱信号”很容易被忽略，最终演变为严重故障，导致非计划停机，增加维护成本，甚至带来较大的经济损失。

本赛题在“工控成套应用”方向中明确提出了“设备故障预警系统”的研究任务，要求基于振动、温度、电流等多源数据，结合 LSTM、SVM、随机森林等算法，实现对设备异常状态的提前识别。同时，初赛阶段需通过可研报告、算法流程、伪代码以及仿真结果来验证方案的可行性。

基于上述背景，本文选取工业循环水泵驱动电机及其轴承系统作为研究对象，设计一套面向工业现场的多源传感融合故障预警系统，并预留与施耐德 EAE 开放自动化平台对接的扩展能力，以满足后续系统落地与拓展需求。

# 1.2 行业痛点与实际需求

从当前工业现场的实际情况来看，旋转设备的运维管理仍然存在一些典型问题，

可以概括为“发现滞后、判断粗糙、联动不足”。

首先，在很多场景中，设备维护仍然依赖人工巡检或事后维修。当设备已经出现明显异常（如异响、温升或性能下降）时才进行处理，这种方式难以满足连续生产对高可靠性的要求。其次，一些已有系统虽然具备基本的温度或电流报警功能，但大多采用固定阈值判断。这种方式难以适应负载变化和工况切换，对早期微弱故障识别能力不足，容易出现误报或漏报。并且振动、温度、电流等数据往往分散在不同系统中，缺乏统一的融合分析机制，导致设备健康状态难以被准确评估。此外，现有报警系统通常只提供“是否异常”的提示，缺乏与控制系统的深度联动，无法进一步支持如降载运行、设备切换或维护决策等操作。

在工业循环水泵场景中，这些问题尤为突出。例如，当电机出现轴承磨损、叶轮不平衡或空化引起的振动异常时，如果不能在早期进行识别与干预，不仅会降低运行效率，还可能引发系统级停机。

因此，现场亟需一种集“在线监测 $^ +$ 数据融合 $^ +$ 智能预警 $^ +$ 控制联动”于一体的解决方案：一方面实现多源数据的持续采集与综合分析，动态评估设备健康状态；另一方面能够根据预警结果进行分级处理，并在 HMI 界面中直观展示，同时为 PLC 和上位系统提供联动接口。

这一需求与赛题倡导的开放自动化理念高度契合，即算法分析与控制执行解耦，通过标准通信实现协同工作。

# 1.3 项目研究目标

本项目围绕工业循环水泵电机故障预警问题，重点提升系统的早期识别能力、预警准确性以及控制联动能力。具体目标如下：

# （1）构建多源状态监测体系

围绕振动、温度、电流等关键参数，建立统一的数据采集与状态表征方法，实现对设备运行状态的连续感知。

# （2）设计数据处理与特征提取流程

包括信号滤波、归一化、分段处理以及时域和频域特征提取，为模型训练提供高质量输入数据。

# （3）建立故障预警模型体系

以 SVM 和随机森林作为基线模型，引入 LSTM 处理时间序列特征，实现对正常、异常及故障状态的识别与预测，重点提升早期故障检测能力。

# （4）设计预警分级与控制策略

将模型输出转化为分级预警结果，并结合实际工况制定对应的报警提示和控制联动策略，形成完整闭环。

# （5）完成仿真验证与方案落地准备

基于MATLAB/Simulink 进行建模与仿真分析，输出算法流程、伪代码及实验结果，为后续系统实现提供依据。

整体目标与赛题分阶段要求相一致：初赛侧重方案验证，复赛推进系统实现，决赛完成完整系统展示。

# 1.4 项目总体思路

本项目总体采用“数据驱动 $^ +$ 分层架构 $^ +$ 闭环控制”的设计思路。

在对象选择上，以工业循环水泵电机及轴承系统为核心，同时兼顾对其他旋转设备的可扩展性。

在数据处理上，以振动、温度和电流信号为输入，通过滤波、去噪、归一化及滑动窗口分割等步骤完成预处理，并提取反映设备状态的多维特征。

在算法设计上，先构建 SVM 和随机森林等基线模型，再引入 LSTM 模型增强对时间序列特征的建模能力，通过多模型对比选取综合性能最优方案。

在决策层面，将设备状态划分为“正常—关注—预警—故障”等等级，并设计对应的响应策略。

在展示层面，通过 HMI 或上位机界面实现设备状态和预警结果的可视化。

系统架构采用“软件分析侧 $^ +$ 控制执行侧”的分层设计：

软件侧负责数据分析与模型推理，控制侧负责数据采集、控制执行及界面显示。两者通过 MQTT 协议进行通信，采用 JSON 格式传输数据，并基于 Mosquitto 构建消息通道，从而实现完整的数据闭环。

# 1.5 预期成果形式

在初赛阶段，本项目的成果主要体现在以下几个方面：

第一，完成一份结构完整、逻辑清晰的可行性研究报告，系统说明项目背景、技术路线及创新点。

第二，输出系统架构图、数据流图、算法流程图及控制逻辑图等关键设计文档。

第三，基于 MATLAB/Simulink 完成算法仿真验证，包括信号处理结果、特征分析结果及模型对比结果，并提供核心算法伪代码。

第四，提出面向后续阶段的扩展方案，包括 MQTT 通信设计、数据格式定义及控制联动逻辑，为系统落地做好准备。

总体来看，初赛阶段重点验证“方案可行性”，后续逐步过渡到“系统实现与现场演示”，与赛题要求保持一致。

# 第二章 赛题理解与需求分析

# 2.1 赛题要求解读

本赛题聚焦“工业控制智能应用”，其核心目标并非单纯完成某一传统控制功能，而是围绕工业控制产品与应用场景，结合人工智能、物联网、边缘计算等新技术，针对真实工业问题提出具有创新性与可实现性的解决方案。赛题明确提出，参赛作品应从给定研究方向中选择一个方向进行研究，并鼓励参赛团队在选题后围绕某一关键指标提出创新性解决思路，而不要求同时实现全部性能指标。这意味着项目设计应坚持“聚焦真实问题、突出核心创新、兼顾工程落地”的总体思路，避免方案过于分散或仅停留在概念描述层面。

从研究方向设置来看，赛题共包含无线传输应用、低谐波整流/回馈单元有源前端以及工控成套应用三大类。其中，工控成套应用重点强调面向实际工业自动化控制中的具体问题提出解决方案，并实现原型系统；在该方向下，题目进一步给出了 AI视觉缺陷检测、污水处理药剂投加智能控制、设备故障预警系统、智能调度与产线协同控制系统等子方向。尤其值得注意的是，赛题对“设备故障预警系统”给出了较明确的技术边界，即基于振动、温度、电流等传感器数据，构建故障预警模型，模型形式可采用 LSTM、SVM、随机森林等，以实现对设备异常状态的提前预判并减少非计划停机时间。这说明赛题本身已经为本项目的监测对象、数据来源、模型类型和应用目标提供了直接指引。

此外，赛题在“开放自动化应用”部分提出了较完整的工程实现规范：系统采用双层开发架构，软件侧需在 Debian 12 环境中独立开发算法模块、数据分析服务和AI 模型推理引擎，不直接操作硬件，而是通过 MQTT 协议与 EAE 平台交换数据；

EAE 侧则负责现场 I/O 控制、PLC 程序和 HMI 显示，数据格式建议使用 JSON，消息代理推荐使用 Mosquitto。虽然本项目当前选择的是“设备故障预警系统”方向，但这一开放自动化要求为后续系统的工程化落地提供了可直接参考的技术路线，也为项目在复赛和决赛阶段向“可部署、可通信、可展示”的完整原型演进提供了明确边界。

从提交要求看，赛题对不同阶段成果有清晰的递进式安排。初赛阶段要求提交可研报告，内容需覆盖选题背景、行业痛点、技术路线图，并通过文献综述或调研报告证明方案可行；若方案中包含算法，还需给出算法流程图、伪代码以及仿真结果，如波形图、误差曲线或准确率对比等。复赛阶段要求在初设报告中给出理论预期与初步实验结果的对比分析，形成包含原理框图和控制逻辑图在内的完整理论设计框架，并配套完成度达到 $7 0 \%$ 的实物作品。决赛阶段则进一步要求提供完整技术方案、软件流程、关键算法公式与参数设定、仿真与实测对比、经济性评估、市场前景与推广路径分析，并提交可现场演示的最终实物系统。由此可见，赛题评价重点不仅在于算法性能本身，也在于系统方案的完整性、工程实现能力及后续推广价值。

# 2.2 研究方向与本项目对应关系

结合赛题要求、团队专业背景以及已收集的参考资料，本文选择“工控成套应用”中的“设备故障预警系统”作为研究方向，并进一步将应用对象聚焦为工业泵房场景下的循环水泵驱动电机及相关配电设备。这样的选题与赛题要求具有较高的一致性：一方面，泵房、水泵、电机和配电系统属于典型工业控制对象，具备明确的工艺场景和工程需求；另一方面，泵房系统中的故障诊断与状态预警本身具有显著的现实价值，设备故障会直接影响供水稳定性和系统安全性，因此非常适合作为“真实工业问题”进行建模与分析。

# 第三章 国内外研究现状与文献综述

# 3.1 工业设备故障诊断/预警技术发展现状

随着工业自动化、智能制造和预测性维护理念的发展，工业设备状态监测与故障预警技术正由传统的“事后维修”与“定期检修”模式，逐步向基于状态评估的在线监测和智能预警方向演进。现有研究普遍认为，传统维护模式存在响应滞后、资源浪费和个体差异难以体现等问题，难以满足现代工业场景对高可靠性、低停机率和低运维成本的要求。尤其在机电设备长期连续运行的场景中，单纯依赖故障停机后修复或固定周期检修，往往难以及时识别潜在故障，也无法为维护决策提供足够的提前量。

从监测对象与感知方式来看，当前研究正在从单一传感器监测向多源/多模态融合监测演进。已有资料表明，传统 CBM 系统通常依赖单一类型的传感器数据，如仅使用振动信号进行分析，但某些故障在振动特征尚不明显时，可能已在温度、声学或电气参数上有所体现。因此，围绕振动、温度、电流、电压、声学、热成像等多源信息构建综合感知系统，已成为工业设备健康监测的重要发展方向。多传感器融合理论也指出，信息融合的本质是对不同传感器采集的数据进行多层次、多空间的信息互补与优化组合，经过特征提取、模式识别和融合决策后，形成对目标状态的一致性解释，从而提升系统的智能化水平。

从具体工程应用看，国内研究已经逐步从“监测”走向“诊断”和“预警”。以泵房设备为例，已有研究利用振动、压力、温度、电压、电流、功率等参数，对轴承故障、水泵汽蚀、吸水口堵塞、泵轴变形等问题进行综合分析；进一步的研究则开始关注配电设备与电机本体故障的在线诊断，如基于多点电压与电流信息融合，对供

电电压超限、空气开关或接触器开路、电机绕组开路、电机绕组匝间短路、水泵堵转以及电机与水泵连接机构断开等故障进行识别，并通过 MATLAB/Simulink 仿真验证其可行性。这说明工业故障诊断研究已经从经验性分析逐步发展到具有明确机理建模、在线判据和仿真支撑的阶段。

从系统架构层面看，当前研究又进一步向“边缘计算 $^ +$ 实时通信 $^ +$ 智能算法 +可视化”一体化方向发展。你提供的资料中，多项方案都采用了较为清晰的分层结构：前端通过振动、温度、声学或电气传感器实现多源采集；边缘侧完成去噪、滤波、缺失值处理、格式统一和初步异常检测；平台侧再进行时间同步、空间对齐、特征提取和特征融合；在此基础上，利用 LSTM、随机森林、自编码器、CNN-LSTM 等模型进行趋势预测、故障分类或无监督异常检测，并借助 MQTT、Kafka、数字孪生等模块实现数据流转和可视化展示。由此可见，当前故障预警研究的重点已不再局限于“能否检测到异常”，而是更加重视“能否实时、准确、持续、可解释地完成健康评估与预警决策”。

从国际前沿趋势看，多模态融合已成为工业异常检测的重要方向。以 MulSen-AD 为代表的最新研究表明，单一传感器在工业异常检测中存在天然“盲区”：RGB图像擅长识别表面缺陷，但对内部损伤和结构异常感知能力有限；红外模态更适合热异常；3D 点云更适合几何变形检测。该研究通过构建融合 RGB、红外和点云的多传感器异常检测数据集，并设计基准模型对不同传感器组合进行对比，验证了多传感器融合方案在检测性能上显著优于单传感器和双传感器方案。这一结果虽来自工业视觉场景，但其核心结论同样适用于机电设备预警领域，即：复杂工业异常通常无法被单一感知通道完整刻画，多源融合是提升鲁棒性和泛化能力的必然趋势。

# 3.2 传统阈值法与智能算法对比

传统阈值法是工业现场最早也是应用最广泛的故障判断方法之一，其核心思想是针对某一单变量或少量特征量设置报警上限、下限或区间判据，当监测值越界时触发告警。此类方法的优势在于实现简单、逻辑清晰、解释性强，尤其适用于具有明确物理边界的故障场景。例如，在泵房配电系统诊断中，供电电压超限可以通过电压偏差阈值进行判断，电流不平衡可通过不平衡率判据识别，水泵堵转与电机连接机构断开也可通过平均电流大小规律给出诊断条件。这类方法建立在机理分析和电气量判据之上，适合工程初期快速落地，也便于在 PLC 或控制逻辑中直接实现。

然而，传统阈值法的局限性也十分明显。首先，阈值通常依赖经验设定或固定规则，难以适应负载变化、工况切换、环境扰动和设备老化带来的参数漂移。其次，单阈值或少量规则往往只能反映单一变量的异常，难以处理多参数耦合、非线性演化和早期弱故障问题。已有研究指出，固定阈值报警系统误报率较高，难以识别早期潜在故障；多参数监测数据即便已被采集，如果仍采用独立分析方式，也无法充分利用参数间的关联信息；依赖经验寿命曲线或简单线性预测的方式，同样难以适应实际设备的个体差异与复杂工况变化。

相比之下，智能算法的优势在于能够处理多变量输入、非线性关系和时序演化过程。在多传感器融合框架下，智能算法可以先对振动、温度、电流、电压、声学等信息进行统一表征，再通过分类、预测或无监督学习实现健康评估和故障识别。例如，LSTM 更适用于对健康指标进行时间序列建模，提前预测性能衰退趋势；随机森林更适合多维特征下的故障类型分类，并能够评估特征重要性；自编码器则适合在缺少异常样本时，通过重构误差发现偏离正常模式的异常行为。与传统阈值法相比，智能算法更适合用于复杂工况下的综合诊断和提前预警。

但需要指出的是，智能算法并非完全替代阈值法。对于电压超限、电流开路、过流等具有明确物理约束的故障，阈值法仍然具有响应快、解释强、便于控制联动的优势；而对于弱故障、复合故障和趋势性衰退问题，智能算法的表现通常更优。因此，当前较为合理的技术路线并不是“二选一”，而是采用机理判据 $^ +$ 数据驱动模型相结合的方式：对显性故障采用阈值或规则快速筛查，对复杂和隐性异常则引入机器学习或深度学习进行进一步识别与趋势预警。这种思路也更符合工业控制场景对可靠性和可解释性的双重要求。

# 3.3 常见模型对比分析

# 3.3.1 SVM

支持向量机（SVM）是一类经典的监督学习模型，适合处理中小样本、高维特征和非线性分类问题。在工业设备故障诊断中，SVM 通常与人工提取的时域、频域或时频域特征相结合，用于实现正常/故障状态区分或多故障类别识别。由于 SVM 对特征维度适应性较强，且在样本量不大时通常具有较好的泛化能力，因此在初期研究和基线模型构建中具有较高实用价值。赛题也明确将 SVM 列为设备故障预警系统可采用的代表模型之一，说明其在该类应用中具有较好的适配性。

对于本项目而言，SVM 的主要优势在于：一是适合初赛阶段基于 MATLAB 提取特征后快速建立分类基线；二是模型结构相对清晰，便于进行不同特征组合的效果比较；三是在样本规模有限、仿真数据占比较高的情况下，通常比深层网络更容易训练稳定。其不足在于：SVM 更偏向静态分类，对时间依赖信息利用不足，当设备状态呈现连续退化趋势时，仅用 SVM 难以反映预警提前量和状态演化过程。因此，SVM更适合作为本项目的基线分类模型，用于验证多源特征的区分能力，而不宜单独承担趋势预警任务。

# 3.3.2 随机森林

随机森林（Random Forest）是一种集成学习方法，通过构建多棵决策树并综合其结果完成分类或回归任务。你提供的材料明确指出，随机森林具有准确率高、鲁棒性强、不易过拟合等优点，尤其适用于多模态特征融合后的故障分类任务；同时，它还能评估不同特征对分类结果的贡献度，这对于工程人员理解故障机理和筛选关键变量具有现实价值。

在工业设备状态监测中，随机森林相较于单一决策树更稳定，相较于深度网络又对样本规模要求更低，因而非常适合作为多源故障诊断中的主力分类模型之一。对于本项目这类基于振动、温度、电流/电压构建多维特征向量的场景，随机森林能够较好地完成正常状态、绕组故障、堵转故障、连接断开等类别的识别，并可用于比较不同特征在诊断中的重要程度。其不足主要在于：随机森林擅长“状态分类”，但对连续时间序列中的长期依赖关系建模能力有限，因此若目标是提前预测未来健康趋势，仍需与 LSTM 等时序模型结合使用。基于此，随机森林较适合作为本项目的主分类模型，与 SVM 形成对比验证。

# 3.3.3 LSTM

长短期记忆网络（LSTM）是处理时间序列问题的典型深度学习模型，能够有效捕捉长期依赖关系。已有材料明确指出，LSTM 模块可基于多模态时序特征数据对健康指标进行预测，通过比较未来一段时间的预测结果与安全阈值，实现趋势预警；其优势在于能够提前发现性能衰退迹象，为预测性维护争取时间窗口。赛题也将LSTM 作为设备故障预警系统的推荐模型，说明该模型与项目目标高度吻合。

相较于 SVM 和随机森林，LSTM 最大的优势在于它不是简单判断“当前是否故障”，而是能够利用历史数据刻画“未来是否正在走向故障”。这使其更符合“预

警”而非“诊断”的要求。对于泵房设备、电机和轴承这类具有渐进退化特征的对象，LSTM 能够利用振动 RMS、温度变化、电流趋势等时序指标，对健康状态的演化进行建模。不过，LSTM 对数据量、样本连续性和训练质量要求更高，若数据规模偏小、标注不充分或工况变化剧烈，模型训练稳定性会受到影响。因此，在本项目中，LSTM更适合作为趋势预测模型，与 SVM/随机森林形成“分类 $^ +$ 预测”的组合方案。

# 3.3.4 其他可选模型

除上述三类模型外，当前研究中还有若干模型路线值得关注。其一是自编码器（Autoencoder），该方法通过学习正常运行数据的重构表示，以重构误差作为异常判别依据，适用于异常样本较少或早期故障标签不足的场景。相关资料中已将自编码器作为多模态工业设备健康监测系统中的重要组成部分，用于实现早期故障的无监督检测，这对本项目未来从“有监督故障分类”扩展到“无监督异常预警”具有参考价值。

其二是 CNN-LSTM 等混合模型。相关专利资料提出，可先通过 CNN 对健康指标和历史数据进行特征提取，再结合 LSTM 进行时间序列趋势建模，从而实现对故障发展态势的推演，并驱动动态阈值优化。这类模型兼具局部特征提取与时序预测能力，适合在后续阶段进一步提高预警性能。

其三是图神经网络与注意力机制模型。在更复杂的关键部件预测任务中，研究开始利用设备物理连接关系构建动态拓扑图，通过图神经网络量化故障传播路径与强度，并引入多头注意力、动态邻接矩阵和知识图谱等机制，实现从局部状态到整体健康度的多层次表征。对于泵房系统这类存在“配电回路—电机—水泵—负载”耦合关系的设备群体，这一路线具有较强的拓展潜力，但实现复杂度较高，更适合作为复赛或决赛阶段的增强方案。

其四是卷积网络及其变体。你提供的资料中，还展示了面向多源轴承故障诊断的 VGG、ResNet、DRSN 等模型，说明在多通道信号分类任务中，深层卷积网络也是常见路线。这类模型在样本充足、数据结构稳定时往往具有较好的识别能力，但其训练依赖更大规模的数据集，且对可解释性和工程部署提出更高要求。对本项目初赛阶段而言，可将其视为方法储备，而不宜作为首选主线。

综合来看，若从比赛实现难度、数据条件和工程可落地性出发，较合理的模型配置应为：SVM 用于基线分类对比，随机森林用于多特征主分类，LSTM 用于趋势预警；自编码器、CNN-LSTM、图神经网络等作为后续增强方案。 这种组合既满足赛题要求，也便于从初赛平滑延展至复赛和决赛。

# 3.4 当前研究的不足

尽管现有研究已经在工业设备故障监测、泵房故障诊断和多源信息融合方面取得了较多进展，但仍存在若干不足。首先，单传感器或单变量分析仍然普遍存在。无论是在机电设备监测领域，还是在工业视觉异常检测领域，单一感知通道都存在天然盲区，难以完整反映复杂异常的不同维度特征。对于机电设备而言，仅看振动可能漏掉早期热异常或电气异常；对于产品异常检测而言，仅看 RGB 图像也难以发现内部裂纹或几何形变。

其次，多源数据虽已采集，但高质量融合仍是难点。资料中多次提到数据异构性、采样率差异、时间不同步、空间映射不一致、噪声与异常值污染等问题。若这些问题得不到有效解决，即使引入更复杂的模型，也可能因输入质量不足而导致预警失真。因此，时间同步、空间对齐、去噪、重采样和特征融合，不是附属步骤，而是影响最终性能的关键环节。

再次，很多研究更偏重“诊断”，而非“预警”。已有方法能够识别故障发生后的类别差异，但对于设备由正常状态向异常状态的退化过程建模仍不充分。尤其是在泵房系统中，部分故障仍主要依赖人工排查，缺少对开关开路、绕组开路、堵转等故障的持续在线预警机制。赛题所强调的“提前预判异常状态、减少非计划停机”，恰恰要求模型不仅能分辨故障类型，还要能够捕捉趋势变化并提前触发控制响应。

最后，系统闭环与持续学习能力不足也是当前研究的共性问题。许多方案停留在离线识别或单次训练阶段，缺少与边缘计算、MQTT 实时通信、HMI、数字孪生以及控制执行的深度耦合，也缺少利用历史维修案例和新故障样本持续优化模型的机制。相关资料已提出基于知识引导、自学习和图神经网络的持续优化方向，但这类能力在多数实际工程方案中尚未完全成熟。

# 3.5 本项目可行性结论

综合上述文献与调研资料，可以认为本项目具备较强的技术可行性和工程实现基础。首先，从赛题要求看，设备故障预警系统本身就是官方明确给出的研究方向，且直接指定了振动、温度、电流等传感器数据以及 LSTM、SVM、随机森林等模型路线，这与本项目拟采用的技术方案完全一致。

其次，从应用场景看，泵房设备故障诊断已有较明确的工程研究基础。基于多点电压电流信息融合的泵房设备故障诊断方法已经对典型故障机理、诊断判据和MATLAB/Simulink 仿真流程进行了系统分析，并验证了其在泵房场景中的有效性和通用性。这意味着本项目以“泵房水泵驱动电机及配电系统”为对象开展预警研究，并非空泛设想，而是建立在已有工程方法和仿真基础之上的进一步拓展。

再次，从方法路径看，多传感器融合理论、工业健康监测系统架构和多源故障预测方法已经为本项目提供了清晰的技术主线，即：多源采集 时间同步/空间对齐

→ 特征提取与融合 分类诊断/趋势预测 动态阈值与预警联动 可视化展示。这条路径既有理论支撑，也有工程案例支撑，且能够与赛题后续要求的 MQTT 通信、边缘计算、EAE/PLC/HMI 联动自然衔接。

因此，可以得出结论：本项目选择“基于多源传感融合的泵房设备故障预警系统”作为研究对象，采用“机理判据 $^ +$ 随机森林/SVM 分类 + LSTM 趋势预测”的总体路线，不仅符合当前国内外研究发展方向，也具备较好的实现基础、比赛适配性和后续扩展潜力，可作为本赛题初赛阶段的可行方案。

# 第四章 项目总体方案设计

# 4.1 总体架构设计

本项目面向工业泵房场景下的循环水泵驱动电机及其配电回路，围绕“多源感知—数据融合—智能预警—控制联动—界面展示”的主线，构建一套分层、模块化、可扩展的工业设备故障预警总体架构。结合赛题要求与现有多源故障预警系统研究成果，系统总体采用“五层结构 $^ +$ 双侧部署”的设计思想：底层为传感采集层，中间依次为数据处理层、预警算法层、控制执行层，顶层为 HMI 显示层；从部署角度看，前三级主要归属于软件分析侧，后两级可逐步与 EAE 控制侧对接，形成符合开放自动化要求的双层架构。

在监测对象上，本项目以泵房水泵电机、轴承及配电主回路为核心对象，结合已有泵房故障诊断研究中对供电电压超限、空气开关或接触器开路、电机绕组开路、绕组不平衡与匝间短路、水泵堵转以及电机与水泵连接机构断开等典型故障的分析结果，确定本系统的多源输入以电压、电流、振动和温度为主。其中，电压电流信息用于反映供电质量、负载变化和电机电气故障；振动信息用于感知轴承、转子和联轴器等机械异常；温度信息用于反映过热、过载和润滑不良等状态。这样的输入配置既贴合赛题对“振动、温度、电流”等传感器数据的要求，也与泵房应用场景高度一致。

从信息处理逻辑看，本项目总体架构遵循多传感器融合的一般原理，即先完成多源数据采集，再进行时间同步、空间/对象对齐、特征提取与模式识别，最后通过融合决策形成对设备状态的一致性解释。现有多传感器融合理论表明，时间同步、空间同步以及前融合/后融合机制，是保证多源数据有效利用的基础。因此，本项目在总体

设计中不把“算法”视为唯一核心，而是将数据质量控制、同步校准和特征融合同样作为关键组成部分。

基于上述思路，本项目的总体架构图可概括如下：

传感采集层

（振动传感器 / 温度传感器 / 电流传感器 / 电压采集节点）

↓

↓ 

预警算法层

（SVM基线分类 / 随机森林主分类 / LSTM趋势预测 / 动态阈值生成）

↓

（预警分级 / 报警联动 / 负载调整建议 / EAE-PLC-I/O 执行）

# 4.2 系统功能模块划分

基于上述总体架构，本项目将系统划分为六个核心功能模块，分别是多源传感采集模块、边缘预处理模块、特征工程与状态建模模块、故障预警算法模块、控制执

行与联动模块以及 HMI 显示与交互模块。该划分方式一方面吸收了已有多源工业健康监测系统中“采集—预处理—融合—预测—决策—可视化”的典型结构，另一方面也兼顾了本项目当前初赛阶段以仿真和方案设计为主、后续阶段可扩展到在线采集和现场联动的实施需求。

# （1）多源传感采集模块

该模块负责完成设备运行数据的原始获取，是整个系统的数据入口。根据泵房设备工况和故障类型，本项目拟配置振动、温度、电流和电压四类主要输入。其中，电压电流信号可由智能电量采集装置或等效采集模块获取，振动数据由安装在电机轴承座或机壳关键位置的加速度传感器获取，温度数据可由贴片式温度传感器或热成像方式获取。现有多源预警系统研究也表明，振动、电气参数和热状态的组合能够较全面反映机电设备的运行状态。

# （2）边缘预处理模块

该模块主要负责对采集到的原始数据进行初步筛查和标准化处理，包括错误数据剔除、格式统一、时间同步、异常过滤和基础去噪等。已有多源传感预警系统明确指出，边缘侧完成初筛、格式统一、时间同步与异常信号过滤，有助于降低后续平台处理负荷并提高输入质量；多模态工业健康监测方案也强调，边缘网关可承担去噪、滤波、缺失值填充、数据格式转换和初步异常检测等任务，以实现快速响应。对本项目而言，该模块既可在 MATLAB 离线处理中实现，也可在复赛阶段迁移到工控机、边缘计算节点或 Debian 软件服务中运行。

# （3）特征工程与状态建模模块

该模块在边缘预处理之后，对多源数据进行深度清洗、特征提取、融合建模和健康指标构建。依据已有研究，可从时域、频域、时频域和统计域等多个维度提取特

征，如均值、方差、RMS、峭度、频谱能量、温度梯度、电流不平衡率等；同时，还需完成多源数据的时间同步和对象关联，形成统一的多维特征向量。进一步地，系统可在此基础上构建健康指标或状态表征，为后续预警模型提供结构化输入。

# （4）故障预警算法模块

该模块是系统实现“异常识别 $^ +$ 趋势预警”的核心。结合赛题要求与前文模型对比，本项目采用“基线分类 $^ +$ 主分类 $^ +$ 趋势预测”的组合思路：以 SVM 作为基线分类模型，用于验证多源特征的区分效果；以随机森林作为主分类模型，用于完成不同故障类型识别；以 LSTM 作为趋势预测模型，用于对设备健康状态进行时间序列建模，提前预判潜在性能衰退。此外，模块还可叠加动态阈值机制，使模型输出结果转化为分级预警依据。现有多源实时健康监测系统研究中，LSTM、随机森林和自编码器的组合已被用于时序预测、分类诊断和异常检测，说明本项目的模型配置具有较强可行性。

# （5）控制执行与联动模块

该模块负责将算法输出的故障概率、健康评分和预警等级转化为具体控制动作或运行建议。对于初赛阶段，该模块主要体现为逻辑设计，包括预警等级划分、报警触发条件、维护提示和降载/停机建议；对于后续复赛、决赛阶段，则可进一步与EAE/PLC 侧联动，实现声光报警、继电器切换、变频调速、负载调整或备用泵切换等动作。赛题要求开放自动化应用能展示“数据处理 决策输出 控制执行 ${ \bf \Gamma } \to \mathrm { H M I }$ 反馈”的全流程，因此该模块是未来系统落地不可缺少的一环。

# （6）HMI 显示与交互模块

该模块承担系统运行状态展示、告警推送、趋势曲线显示和人工交互功能。现有多源工业健康监测系统通常通过数字孪生或可视化界面实现状态展示与交互式查询；赛题也明确要求 EAE 侧完成 HMI 显示任务。因此，本项目在设计上将 HMI 层作为独

立模块，主要展示设备实时状态、关键特征量、故障类别、健康度趋势、预警级别及维护建议，并为后续 WEB/APP 或上位机扩展保留接口。

# 4.3 工作流程说明

本项目的工作流程遵循“采集—处理—识别—决策—执行—反馈”的闭环逻辑，与多传感器融合的一般处理流程及赛题对开放自动化应用的全链条要求保持一致。其核心目标不是简单输出一个故障类别，而是形成一套从底层信号到上层控制与界面展示的完整工业控制智能应用流程。

在采集阶段，系统通过振动、温度、电流、电压等传感节点对泵房设备进行持续监测，获取表征机械状态、电气状态和热状态的原始数据。由于泵房系统故障不仅表现为单一机械异常，还涉及供电、负载和配电回路变化，因此多源采集是保证状态感知完整性的前提。泵房故障诊断研究已表明，关键节点的电压、电流变化可反映供电超限、接触器开路、绕组开路、水泵堵转等多类故障。

在数据处理阶段，系统对原始数据进行时间同步、格式统一、去噪清洗和特征提取。这里既包括对不同采样率数据的时间戳校准，也包括对信号噪声、异常点和环境漂移的修正。已有研究建议通过统一时间戳、插值或重采样实现时间同步，并通过小波去噪、Z-score 标准化等方式完成质量优化。完成这些处理后，系统再从时域、频域、时频域和统计域提取特征，形成面向算法层输入的多维特征向量。

在预警算法阶段，系统先利用 SVM 与随机森林对当前状态进行分类判断，识别是否存在供电异常、绕组故障、堵转等典型问题；再利用 LSTM 对健康指标变化趋势进行预测，得到未来短时间内的异常概率或故障风险。在这一基础上，系统引入动态阈值或分级规则，将模型输出映射为“正常—关注—预警—报警”等不同等级，以增强预警的自适应性并降低固定阈值导致的误报漏报。

在控制执行阶段，系统依据预警等级和故障类型输出相应的处理结果。对于轻微异常，可提示运维人员进行巡检；对于持续恶化趋势，可建议降载运行或计划检修；对于严重异常，则可触发报警并联动控制逻辑。在赛题要求的开放自动化扩展中，这一阶段可由 EAE/PLC 完成现场 I/O 执行与逻辑响应，而软件侧仅负责分析结果发布。

在 HMI 显示与反馈阶段，系统将实时状态、关键特征、预警等级和历史趋势以图形化方式呈现给操作人员，同时记录日志和决策结果，为后续模型优化与运维复盘提供依据。如果引入数字孪生或知识反馈机制，还可将执行结果反向输入模型层，实现“监测—分析—决策—优化”的闭环迭代。

结合本项目的应用对象、赛题要求和已有研究基础，本文将关键技术路线概括为“多源感知、分层处理、融合预警、闭环控制、开放扩展”五个环节。该技术路线既能够满足初赛阶段以方案论证和仿真验证为主的任务要求，也能够自然延展到复赛和决赛阶段的原型实现。

第一步是多源感知与对象建模。围绕泵房电机及其配电回路，布置振动、温度、电流、电压采集节点，构建覆盖机械、电气和热状态的感知体系；同时，依据泵房配电系统结构和典型故障机理，明确监测对象、故障类型和关键观测量，为后续状态建模打下基础。

第二步是数据预处理与同步校准。对采集到的多源异构数据进行清洗、去噪、格式标准化、时间同步和对象关联。这里重点解决不同传感器采样周期不同、数据格式不同以及环境噪声干扰的问题。已有研究明确指出，时间同步和空间/对象对齐是多传感器融合的关键前提。

第三步是特征提取与融合建模。对同步后的数据提取时域、频域、时频域及统计特征，并构建统一的多维特征向量；在需要时可进一步构建健康指数或状态表征，以增强不同工况下的鲁棒性。对于后续增强方案，还可考虑引入拓扑建模、动态邻接关系和多层次特征融合机制。

第四步是智能预警与动态阈值判定。基于多源特征向量，采用 SVM、随机森林和 LSTM 形成分类诊断与趋势预测的组合算法。再结合动态阈值、规则分级或健康评分机制，实现从“是否异常”到“异常程度与发展趋势”的连续判定。这样既可保留机理判据的可解释性，也可发挥数据驱动模型在复杂场景下的识别能力。

第五步是控制联动与可视化呈现。将算法输出转化为控制建议和告警动作，并通过 HMI 或上位机进行可视化展示；若后续进入开放自动化实现阶段，则将软件分析服务部署在 Debian 12 环境中，通过 MQTT 和 JSON 主题与 EAE 平台通信，由 EAE 侧负责 PLC 程序、现场 I/O 和 HMI 显示，从而形成标准化的软件侧—控制侧协同架构。

# 第五章 数据采集与特征处理方案

# 5.1 监测对象与典型故障类型

本项目的监测对象选取为工业泵房场景下的循环水泵驱动电机及其相关传动、配电单元，重点关注电机本体、轴承与联接机构在连续运行工况下的状态变化。选择该对象的原因在于：泵房系统长期承担连续供水任务，一旦电机或水泵发生故障，容易引起供水不稳定甚至停供，具有明确的工程应用背景；同时，泵房系统的故障机理较清晰，便于结合机理分析与数据驱动方法开展研究。已有研究表明，泵房配电系统和水泵设备中存在多类典型故障，可通过关键节点电压、电流等量的变化进行识别，并已在 MATLAB/Simulink 环境中完成可行性验证。

结合泵房设备故障诊断研究、赛题要求以及本项目计划采用的振动、温度、电流三类传感输入，本文将典型故障类型划分为以下几类。第一类为电气故障，包括电机绕组开路、绕组不平衡与匝间短路等，此类故障通常伴随电流异常、三相不平衡和温升变化。第二类为机械负载故障，包括水泵堵转、联轴器或电机与水泵连接机构断开等，此类故障会引起电机负载突变，并在电流、振动和温度上形成联动特征。第三类为机械健康故障，包括轴承磨损、转子不平衡、对中不良等，虽然泵房故障论文主要从电压电流角度建模，但既有泵房监测研究已证明振动与温度信号对于电机轴承、水泵汽蚀和吸入口堵塞等问题具有较好诊断价值。第四类为热异常状态，主要表现为电机过热、润滑不良或长时间高负载运行导致的温升异常。基于以上分析，初赛阶段可将“正常、绕组开路、绕组不平衡/匝间短路、水泵堵转、连接机构断开、机械异常/轴承异常”作为主要状态类别，为后续模型训练与预警分级提供统一标签体系。

从预警目标看，本项目并不局限于故障发生后的静态识别，而是希望通过多源状态量的持续采集与趋势建模，尽量在故障演化前期就给出风险提示。因此，在故障定义上需要兼顾“明确故障类别”和“状态退化趋势”两个层面：前者服务于分类诊断，后者服务于提前预警。这也与赛题所要求的“基于传感器数据构建故障预警模型，提前预判设备异常状态，减少非计划停机时间”的目标保持一致。

# 5.2 传感器选型与采集信号说

为兼顾系统可实现性、数据有效性和后续扩展能力，本文采用“振动 $^ +$ 温度 $^ +$ 电流”的三源感知方案。其中，振动信号主要反映机械状态，温度信号主要反映热状态，电流信号主要反映电气状态与负载状态。已有多模态工业健康监测研究指出，单一类型传感器往往难以完整表征复杂故障，而振动、温度、电流等多源信息融合能够更全面地反映设备状态，提升故障识别和预警精度。

# （1）振动信号

振动信号是机电设备状态监测中最常用、最敏感的信号之一，特别适用于识别轴承磨损、转子不平衡、对中不良、机械松动以及联接机构异常等机械类故障。已有泵房监测研究采用振动加速度传感器获取振动数据，并结合特征增强与分解方法实现对电机轴承故障、水泵汽蚀和吸水口堵塞等问题的诊断；多模态工业监测方案也明确将不同方向的振动信号作为核心输入之一。基于此，本项目拟在电机轴承座或机壳关键位置布置加速度传感器，采集设备运行中的振动时域波形。对于后续扩展方案，可根据需要增加 X、Y、Z 三方向采集，以提升机械故障感知能力。

# （2）温度信号

温度信号主要用于表征电机绕组、轴承及壳体的热状态，对过载运行、润滑不良、散热异常和潜在热故障具有较好的指示作用。已有多模态传感器融合健康监测研

究将温度传感器作为核心配置之一，用于采集关键部件温度变化；多源预警系统研究还进一步提出，可从温度数据中提取温度极值、温度梯度等特征，作为健康状态表征的重要组成部分。因此，本项目拟在电机壳体、轴承附近或关键热源位置布置温度传感器，用于采集电机表面温度或局部热点温度。初赛阶段可采用接触式温度传感器完成验证，后续若条件允许，也可扩展为热成像或多点温度阵列，以增强热异常监测能力。

# （3）电流信号

电流信号直接反映电机电气状态和负载变化，是泵房设备故障诊断中最具工程价值的信号之一。泵房设备故障诊断研究表明，电机绕组开路、三相不平衡与匝间短路、水泵堵转以及电机与水泵连接机构断开等多类故障，都可通过回路电流大小及其不平衡关系进行区分。比如，绕组开路会导致某相电流降为零，绕组不平衡与匝间短路会造成三相电流不平衡，堵转会引起三相电流整体升高，而连接机构断开则通常导致电机空载、电流明显减小。因此，本项目拟在电机供电回路配置电流传感器，采集三相电流波形，并通过后续特征处理提取均值、有效值、不平衡率及动态变化趋势等指标。

总体上，三类传感器分别覆盖了机械、热、电三种关键状态维度，既符合赛题对“振动、温度、电流等传感器数据”的导向，也兼顾了泵房场景中典型故障的可观测性。该配置能够在硬件复杂度可控的前提下，为后续多源特征融合提供较强支撑。

# 5.3 数据预处理方法

原始传感数据在实际采集过程中不可避免会受到噪声、采样率差异、时间不同步、局部缺失和异常干扰等因素影响，因此在进入特征提取和模型训练之前，必须进行系统的数据预处理。已有多源预警系统研究普遍将边缘侧预处理作为关键步骤，包

括错误数据剔除、格式统一、时间同步、异常过滤、去噪、滤波、缺失值填充和重采样等。本项目在此基础上，重点采用去噪、滤波、归一化和分段四类处理方法，并在此之前完成必要的时间同步和格式统一。

# （1）去噪

去噪的目标是削弱随机干扰、环境噪声和非故障扰动对信号的污染，尽可能保留与设备状态相关的有效信息。针对振动信号，相关研究建议采用小波去噪或变分模态分解等方法，在多尺度下分离噪声与故障特征，以提升弱故障信息的可辨识度；针对温度信号，则可通过滑动窗口平滑和异常值剔除抑制环境波动；针对电流信号，可先去除尖峰和采样毛刺，避免瞬态异常点对后续特征计算造成放大效应。因此，本文拟采用“统计初筛 $^ +$ 小波去噪”为主的处理方案：先利用统计规则识别明显异常点，再通过小波分解对振动和电流信号进行深度去噪，对温度信号则采用滑动平均或滑动窗口异常修正方法。

# （2）滤波

滤波主要用于抑制特定频段的噪声成分并增强故障相关频段的有效信息。多源预警系统中，边缘预处理模块通常会将滤波作为基础处理手段，用于提升后续特征提取质量。对本项目而言，振动信号可根据采样频率和设备转速范围设计带通或低通滤波，突出机械振动有效成分；电流信号可采用低通滤波或工频同步处理，抑制高频干扰并保留负载变化和不平衡特征；温度信号由于变化速率较慢，可采用中值滤波或低阶平滑滤波，降低测量抖动的影响。需要说明的是，滤波参数的选取应结合实验平台、电机额定转速和采样频率进行调整，以保证不过度削弱真实故障特征。

# （3）归一化

由于振动、温度和电流三类信号在量纲、数值范围和变化速度上存在显著差异，若直接融合输入模型，容易导致某一类大尺度特征主导训练过程。因此，归一化是多源特征融合前的必要步骤。已有研究提出，可通过 Z-score 标准化削弱环境因素和量纲差异对模型的影响，也可在需要时采用 Min-Max 归一化将特征压缩到统一范围。结合本项目实际，本文拟对连续特征优先采用 Z-score 标准化处理，使不同信号维度满足零均值、单位方差分布；对于后续部署到边缘侧或需要固定范围显示的特征，也可采用Min-Max 方式将其映射至 [0,1] 区间。

# （4）分段

为了将连续时间序列转化为可供模型训练和推理的样本，本项目采用滑动窗口分段方法对预处理后的信号进行切片。该方法在故障诊断任务中较为常见，可兼顾样本量扩充与时间局部特征保留。你提供的多源轴承故障诊断资料中采用了窗口长度(W=1024)、重叠率 (overlap=0.5) 的设置，并将样本划分为训练集、验证集和测试集，为构建时序分类数据集提供了直接参考。结合本项目初赛阶段的 MATLAB 仿真与实验数据条件，本文拟优先采用固定长度滑动窗口分段方式：对振动和电流等高频信号使用较短时窗进行切分，对温度等低频信号可采用重采样或时间对齐后再并入同一窗口，确保同一数据片段内包含同步的多源状态信息。初始阶段可参考 $\scriptstyle ( \mathsf { W } = 1 0 2 4 )$ 、 $5 0 \%$ 重叠的设置，后续再根据采样频率和模型效果调整窗口长度。

除上述四项之外，本项目还需要在预处理流程中加入时间同步与格式统一。多源传感器融合研究指出，不同类型传感器往往存在采样周期不同、时间戳错位和数据结构不一致的问题，因此应通过统一时间戳、插值或重采样实现时间对齐，并对不同信号格式进行标准化处理。只有在此基础上形成严格对齐的多源样本，后续的多模态特征提取和模型训练才具有实际意义。

# 5.4 特征提取方法

完成预处理后，需要将原始多源信号转化为具有判别力的状态特征。结合已有多源健康监测研究与本项目对象特点，本文从时域、频域和时频域三个层面开展特征提取，并将温度相关统计量、电流平衡性指标和振动动态指标统一纳入多维特征集，以形成适用于 SVM、随机森林和 LSTM 的输入表示。已有研究明确提出，工业设备故障特征提取可从时域、频域、时频域和统计域展开，并对温度极值、温度梯度等热特征进行补充，这为本项目的特征设计提供了直接依据。

# （1）时域特征

时域特征直接来源于信号波形本身，计算简单、工程解释性较强，是故障诊断中的基础特征。针对振动信号，本文拟提取均值、均方根、方差、标准差、峰值、峰峰值、偏度、峭度、波形因子、脉冲因子和裕度因子等，以反映机械冲击、能量变化和波形畸变情况。针对电流信号，则除均值、有效值、峰值外，还重点提取三相平均电流、电流不平衡率、相间差值等特征，用以表征绕组不平衡、匝间短路和堵转等状态；泵房故障论文已经说明，三相电流平均值和不平衡率是判断堵转、绕组不平衡和连接机构断开的重要依据。针对温度信号，本文拟提取平均温度、最高温度、温升速率、温度波动幅值和温度梯度等特征，用于反映热异常和持续过载状态。

# （2）频域特征

频域特征用于揭示信号在不同频率成分上的能量分布，尤其适合捕捉周期性冲击、谐波变化和不平衡特性。已有多源故障预警系统研究指出，可通过频域分析挖掘设备振动信号和电气信号中的频率信息，从而形成关键故障特征。对本项目而言，振动信号的频域特征可包括主频、倍频成分、频带能量、谱峰值、谱重心等，以表征旋转机械的周期性故障；电流信号可提取工频分量、谐波含量、频谱能量分布和不平衡

相关频带特征，以增强对电机电气异常的识别能力；温度信号虽以慢变量为主，但可通过低频频谱分布表征其长期波动趋势。对于初赛阶段，频域分析可通过 FFT 实现，兼顾实现简便性和工程可解释性。

# （3）时频域特征

考虑到泵房设备故障常常呈现非平稳、渐进演化的特点，仅使用纯时域或纯频域特征可能难以全面描述故障过程，因此本项目进一步引入时频域特征。已有资料明确提出，可通过时频分析融合时间与频率维度信息，以更完整地刻画设备状态变化；相关研究还建议利用小波变换、变分模态分解等方法对非平稳故障特征进行增强。结合本项目实际，初赛阶段拟采用短时傅里叶变换或小波包分解构造时频特征，如各频带能量分布、小波包能量熵、时频谱局部能量变化等，以增强对早期弱故障和瞬态异常的感知能力。对于后续决赛阶段，若数据规模与算力条件允许，还可进一步将时频图谱输入深度模型进行自动特征学习。

综合而言，本文最终构建的特征集将由“振动时域/频域/时频域特征 $^ +$ 温度统计/趋势特征 $^ +$ 电流幅值/平衡性/频谱特征”组成，并在特征层面实现多源融合。这样既能服务于传统机器学习模型的输入需求，也可为 LSTM 的时序建模提供连续特征序列。

# 5.5 数据集构建方案

为了保证本项目在初赛阶段具备可执行性，同时兼顾后续复赛与决赛的工程落地需求，本文采用“仿真数据集 $^ +$ 实测数据集”相结合的两阶段数据集构建方案。其核心思路是：在初赛阶段优先利用 MATLAB/Simulink 建立泵房电机及负载故障仿真模型，快速生成带标签的状态数据，用于验证特征提取与模型设计的可行性；在后续阶段再逐步引入实验台架或现场采集数据，对模型进行校正与增强。已有泵房设备故障

诊断研究已经使用 MATLAB/Simulink 搭建单电机泵房配电系统仿真模型，并对绕组开路、绕组不平衡与匝间短路、水泵堵转、连接机构断开等故障进行了模拟验证，这为本项目构建初始标注数据集提供了直接可借鉴的路径。

在第一阶段，构建仿真数据集。具体做法为：基于电机—水泵—负载系统，在正常运行条件下生成基准数据；然后通过参数扰动和故障注入方式，分别模拟绕组开路、绕组不平衡/匝间短路、水泵堵转、连接机构断开以及机械异常等状态，并同步生成对应的振动、温度、电流序列。对所有数据进行时间同步、预处理、滑动窗口分段后，形成带故障标签的样本集合。样本标签可定义为“0-正常，1-绕组开路，2-绕组不平衡/匝间短路，3-堵转，4-连接机构断开，5-机械异常”。若初期难以完整获得三源同步仿真信号，也可先以电流为主线完成故障类别验证，再逐步补充振动和温度通道，实现从单源到多源的数据扩展。

在第二阶段，构建实测数据集。具体做法为：在实验平台或实物样机上部署振动、温度和电流传感器，采集设备在正常运行和典型异常工况下的多源数据。由于真实故障样本往往难以大规模获取，因此可采用“受控工况 $^ +$ 轻微故障模拟 $^ +$ 长时间运行采样”的方式积累数据，并通过人工标注、运维记录或阈值规则辅助完成标签确认。对后续难以直接采集的故障类型，可继续保留仿真数据作为补充，从而形成“仿真增强 $^ +$ 实测校正”的混合数据集。该方案既符合比赛节奏，也有利于提升模型对真实工况的适应性。

在样本组织上，本文拟采用滑动窗口方式生成样本，并保持各类别样本数量尽可能均衡，以避免类别偏置影响模型训练。参考你提供的多源故障诊断资料，初步可采用窗口长度 $\scriptstyle ( \mathrm { W } = 1 0 2 4 )$ 、重叠率 (overlap=0.5) 的设定，并按照训练集、验证集、测试集 $= 7 { : } 2 { : } 1$ 的比例进行划分。对于温度信号等低频变量，可先通过重采样对齐到统一时

间轴后再进入滑窗；对于样本量不足的类别，可通过窗口平移、噪声微扰、幅值缩放等方式进行适度数据增强。

从用途划分上，数据集将服务于三个层次的模型任务。第一类是面向 SVM 和随机森林的故障分类数据集，以窗口级静态特征向量为输入；第二类是面向 LSTM 的趋势预测数据集，以连续时间窗特征序列为输入，预测未来若干时刻的健康指标或风险等级；第三类是面向后续扩展的异常检测数据集，可在正常样本占主导的前提下支持自编码器等无监督模型训练。通过这样的多用途数据集设计，可以使同一套采集数据同时支撑分类、预测和异常检测任务，增强系统整体扩展性。

综上，本项目的数据集构建方案并不是一次性追求大而全，而是遵循“初赛先验证、复赛再实测、决赛重融合”的渐进式策略：初赛侧重完成仿真数据构建和算法可行性验证，复赛侧重接入在线采集与样机实测，决赛则在仿真与实测融合数据基础上实现更稳定的预警模型与控制联动系统。这种方案既符合比赛阶段性要求，也能够降低前期实现难度，提高整体方案的可落地性。

# 第六章 故障预警算法设计

# 6.1 算法目标定义

本项目的算法设计面向工业泵房场景下的水泵驱动电机及其相关配电回路，核心目标不是单纯完成“故障后识别”，而是实现“状态识别 $^ +$ 趋势预警 $^ +$ 控制支撑”三位一体的故障预警功能。结合赛题对设备故障预警系统的要求，算法需要能够基于振动、温度、电流等传感器数据，对设备异常状态进行提前预判，并尽量减少非计划停机时间；同时，赛题已明确将 SVM、随机森林、LSTM 等模型列为可采用的代表方法，为本项目算法选型提供了直接依据。

基于此，本文将算法目标具体定义为以下三个层面。

第一，完成设备运行状态的分类识别。针对正常状态、绕组开路、绕组不平衡/匝间短路、水泵堵转、连接机构断开以及机械异常等典型工况，利用多源特征向量实现状态分类，为故障定位和运维判断提供依据。已有泵房设备故障诊断研究表明，这些典型故障可通过关键节点电压、电流变化特征进行区分，而多源监测系统进一步表明，结合振动、温度等信号后可提升复杂故障识别能力。

第二，完成设备健康趋势的提前预警。除静态分类外，算法还需从连续监测数据中提取时间演化规律，对未来短时间窗口内的健康状态进行预测，尽可能在故障显性化之前发出预警。已有工业设备实时健康监测方案中，LSTM 被用于对健康指标进行时序预测，自编码器可用于偏离正常模式的异常检测，这说明趋势建模和异常建模是实现“预警”而非仅“诊断”的关键。

第三，为后续控制执行与 HMI 展示提供结构化输出。算法的最终输出不应只是一个类别标签，而应包括故障类别、故障概率、健康评分、预警等级和趋势结果等

信息，使其能够直接服务于后续的预警分级、PLC/EAE 联动和 HMI 可视化展示。相关多源传感智能预警系统研究中，也普遍采用“多源采集—特征工程—预测模型—动态阈值—决策与远程协同”的架构，这与本项目的总体设计高度一致。

因此，本项目算法总体目标可概括为：基于多源传感融合特征，实现泵房设备故障状态的准确分类、健康趋势的提前预测以及预警结果的工程化输出。

# 6.2 基线模型设计

为兼顾初赛阶段的实现可行性、模型可解释性与后续扩展能力，本文采用“基线分类模型 $^ +$ 提升时序模型”的分层设计思路。在基线模型部分，优先选用 SVM 和随机森林两类传统机器学习模型，分别用于多源特征空间下的分类基准建立与主分类模型构建。赛题已明确将这两类模型列为设备故障预警系统的推荐算法，说明其在本项目场景下具备良好的适配性。

# 6.2.1 SVM 基线模型设计

支持向量机适合处理中小规模样本、高维特征和非线性分类问题，特别适合本项目初赛阶段基于 MATLAB/Simulink 或实验数据提取后的结构化特征分类任务。其基本思路是：将每个时间窗口内提取到的多源特征组成一条样本向量，并通过核函数将其映射到高维空间，在高维空间中寻找最优分离超平面，实现不同故障状态的区分。由于泵房故障诊断任务中很多特征具有明确物理意义，如电流不平衡率、均方根、温升速率、谱能量等，SVM 在此类“人工特征 $^ +$ 中小样本”的任务上通常能够取得较稳定的效果。

本项目中，SVM 主要承担两个作用：一是作为最基本的分类基准模型，用于判断所提取特征是否具备区分正常状态与故障状态的能力；二是作为后续随机森林和LSTM 的比较对象，用以验证更复杂模型是否真正带来性能提升。具体实现上，本文

拟采用径向基核函数的多分类 SVM，并使用一对多或一对一策略实现多故障状态分类。输入为经预处理和归一化后的多源特征向量，输出为各故障类别标签及其判别分值。

# 6.2.2 随机森林主分类模型设计

随机森林是一种基于多棵决策树集成的分类模型，适合处理多维异构特征，并对噪声和特征波动具有较强鲁棒性。已有工业设备实时健康监测方案中，随机森林被用于对融合后的多模态特征向量进行状态分类诊断，能够提升故障识别准确率，同时还能输出特征重要性排序，为工程解释提供依据。

本项目中，随机森林作为主分类模型，输入同样为窗口级多源特征向量，但与SVM 相比，它更擅长处理非线性特征组合和特征间复杂交互关系，更适合用于泵房设备这种电气、机械和热状态相互耦合的分类任务。随机森林输出包括故障类别、类别概率以及关键特征贡献度。其中，类别概率可为后续预警分级提供直接参考，关键特征贡献度则可用于分析某一时刻算法判断主要依据于电流、振动还是温度特征，从而增强模型的工程可解释性。

因此，在基线模型设计中，本文采用“SVM 做基准，随机森林做主分类”的双模型策略：前者用于快速建立故障识别下限，后者用于实现更稳健的多故障分类与重要特征分析。

# 6.3 提升模型设计

虽然 SVM 和随机森林能够较好地完成故障状态分类，但它们本质上更偏向于“当前状态判断”，对设备运行状态的连续演化规律刻画能力有限。为了满足本项目“故障预警”而非仅“故障诊断”的需求，本文进一步引入 LSTM 作为提升模型，用于完成健康趋势建模和预警提前量分析。已有多源工业健康监测系统明确提出，LSTM可基于融合后的多模态时序特征，对设备健康指标进行时间序列预测，提前预警潜在

性能衰退或故障趋势。

# 6.3.1 LSTM 趋势预测模型设计

LSTM 的核心优势在于能够利用时间上下文信息捕捉长短期依赖关系，这使其非常适合处理电机运行过程中渐进式演化的异常状态。对于泵房设备而言，电流波动、振动增强和温升异常往往不是瞬时产生，而是随着负载变化、部件磨损和故障累积逐渐发展。因此，本文不将 LSTM 仅用于故障类别识别，而是将其用于对健康状态的连续建模。

具体设计如下：首先，将每个滑动窗口提取的多源特征向量按时间顺序组成特征序列；然后，将连续若干个时间窗的特征序列输入 LSTM 网络；最后，由 LSTM 输出下一时刻或未来若干时刻的健康评分、风险概率或预警等级。若预测结果连续超过阈值，则触发预警。这样，模型既可反映设备当前所处状态，也可给出未来短时间内的异常趋势判断。

# 6.3.2 其他可选时序模型

除 LSTM 外，若后续数据规模、硬件条件或比赛阶段允许，本文还可考虑以下可选时序模型。

其一，自编码器异常检测模型。已有工业设备健康监测系统中，自编码器用于学习正常运行模式，并通过重构误差识别偏离正常模式的异常数据，适合异常样本不足或需进行早期无监督预警的场景。

其二，CNN-LSTM 混合模型。已有多源智能预警系统方案中，采用 CNN-LSTM 结构对健康指标和历史数据进行联合建模，利用 CNN 抽取局部模式，LSTM 刻画时间演化，用于故障趋势推演和动态阈值优化。这种结构在后续复赛或决赛阶段可作为强化模型方案。

其三，图神经网络或时空注意力模型。已有多源关键部件故障预测方案提出，可根据设备物理连接关系构建动态拓扑图，量化故障传播路径与强度，并结合时空分析与记忆建模进行早期预警。这类方法理论上更强，但实现复杂度较高，不适合作为初赛主线，可作为后续增强方向。

综上，本文在初赛阶段将“LSTM 作为主提升模型”，其余时序模型作为后续增强储备。

# 6.4 模型训练与验证流程

为了保证算法验证的完整性和结果的可复现性，本文将模型训练与验证流程划分为“数据准备—样本划分—模型训练—参数调优—结果验证—预警评估”六个步骤。该流程既符合故障诊断类算法的一般研究范式，也与赛题要求的“算法流程 $^ +$ 仿真结果”提交形式一致。

第一步，数据准备。对振动、温度和电流信号进行时间同步、去噪、滤波、归一化和滑窗分段，并提取时域、频域和时频域特征，形成统一的多源特征样本。对于LSTM，则进一步将连续多个时间窗拼接为时间序列输入。多源故障诊断资料中已经采用了窗口分段、重叠采样和训练/验证/测试集划分的标准流程，这可为本项目提供直接参考。

第二步，样本划分。将数据集按训练集、验证集、测试集进行划分。参考你提供的多源轴承故障诊断资料，可采用 7:2:1 的比例进行均匀划分，保证不同类别在各子集中分布尽可能均衡。

第三步，模型训练。对 SVM、随机森林和 LSTM 分别进行训练。SVM 训练重点在于核函数与惩罚系数选择；随机森林训练重点在于树数量、最大深度和叶节点样本数设置；LSTM 训练重点在于输入时间步长、隐藏层单元数、学习率和批次大小等

参数设置。已有相关资料表明，多模态健康监测系统常采用融合后的多维特征向量训练分类与时序预测模型，本项目沿用这一思路。

第四步，参数调优。利用验证集对各模型进行参数搜索与性能比较。对于SVM，可搜索核函数参数与惩罚因子；对于随机森林，可比较不同树数和深度下的性能；对于 LSTM，可比较不同时间步长和隐层单元数对准确率与预警提前量的影响。参数调优的目标不是单纯追求最高准确率，而是兼顾召回率、误报率和推理时延等工程指标。

第五步，结果验证。使用测试集分别输出分类准确率、召回率、混淆矩阵和误报率，比较 SVM 与随机森林的分类表现；对于 LSTM，则重点验证其对健康趋势或异常风险的预测准确性，并比较不同阈值设置下的预警效果。

第六步，预警评估。将模型输出映射为“正常—关注—预警—报警”等状态等级，并统计不同故障工况下的预警提前量和推理时延。若后续接入 HMI 或控制逻辑，还可进一步验证算法输出能否有效支撑预警联动。

# 6.5 评价指标设计

为了从分类性能、预警能力和工程部署可用性三个维度综合评价模型，本文设计以下评价指标。

# 6.5.1 准确率

准确率用于衡量模型对全部样本分类正确的比例，是最基本的总体性能指标。对于 SVM 和随机森林，准确率能够直观反映不同故障类别识别的整体效果；对于LSTM，若将未来时间窗的风险等级离散化，也可计算趋势预测的分类准确率。赛题明确要求在算法方案中给出准确率等仿真或对比结果，因此准确率是本项目的基础指标之一。

# 6.5.2 召回率

召回率用于衡量某一故障类别被正确检出的比例。在工业预警场景中，召回率通常比单纯准确率更重要，因为漏检真实故障可能直接导致设备继续带病运行。对本项目而言，应重点关注堵转、绕组故障和严重机械异常等高风险类别的召回率，确保关键故障尽可能少漏报。

# 6.5.3 误报率

误报率用于反映模型将正常状态误判为异常状态的程度。工业现场若误报过多，会引起不必要的停机、维护成本上升和运维人员信任下降。已有多源故障预测研究指出，传统固定阈值系统往往误报率较高，因此本文需将误报率作为重要评价指标，用于比较传统规则判据与智能模型在稳定性上的差异。

# 6.5.4 预警提前量

预警提前量用于衡量模型在故障显性化之前提前发出预警的时间长度，是本项目区别于普通故障分类任务的核心指标。对于 LSTM 等时序模型，可将“首次预警时刻”与“故障实际发生时刻”之间的时间差定义为预警提前量。该指标越大，说明模型越能为维护和控制决策争取反应时间。赛题强调的是“设备故障预警系统”，因此这一指标具有很强的针对性。

# 6.5.5 推理时延

推理时延用于衡量模型从接收输入到给出结果所需的时间，是评价模型能否部署到工业现场的重要工程指标。尤其当系统后续接入 MQTT、EAE 或 PLC 联动时，若推理时延过高，将影响预警响应速度。因此，在模型比较时，不能只看准确率，也要考虑推理时延与部署复杂度。已有工业实时健康监测方案之所以引入边缘预处理和分

层架构，本质上也是为了平衡精度与实时性。

因此，本文将评价体系概括为：分类性能看准确率与召回率，稳定性看误报率，预警能力看提前量，工程可用性看推理时延。

# 第七章 控制联动与工业应用设计

# 7.1 报警等级划分策略

为了使故障预警结果能够直接服务于工业现场运维与控制决策，本项目采用分级报警机制，将算法输出的故障类别、故障概率、健康评分和趋势预测结果统一映射为“正常—关注—预警—报警”四级状态。这样的设计一方面能够避免单一阈值造成的频繁误报，另一方面也有利于将不同严重程度的异常对应到不同层次的控制响应。已有多源智能预警系统研究表明，将健康评分、预测趋势和动态阈值相结合，能够更好地反映设备状态的连续演化过程，并为后续决策模块提供结构化输入。

“正常”级表示设备运行状态稳定，各主要特征量处于合理范围内，分类模型和趋势模型均未发现异常征兆，此时系统仅进行常规监测与数据记录，不触发额外控制动作。“关注”级表示当前未形成明确故障，但某些特征量已出现轻微偏移，或趋势模型显示健康状态存在弱退化迹象，例如温升速率略有升高、电流波动增强、振动能量轻微异常等。此时系统主要进行状态提示和趋势跟踪，不直接干预现场运行，仅提示运维人员加强观察。

“预警”级表示算法已识别出较高的异常概率，或 LSTM 预测结果表明未来短时间内故障风险明显增加，但设备仍具备继续运行条件。此时系统应触发明显的界面告警与消息推送，并进入联动准备状态，例如生成检修建议、限制高风险启停操作、提示切换低风险运行模式等。“报警”级则表示系统判定已存在明确故障或高危趋势，例如绕组开路、水泵堵转、电流严重不平衡、持续高温等状态，此时需要触发现场声光报警，并结合故障类型执行联动控制逻辑，以避免设备损坏扩大或引发系统性停机。赛题要求系统最终能够展示完整的控制执行与 HMI 反馈过程，这种四级报警结构更便

于实现控制动作与界面显示的一一对应。

从判定机制上看，本文不采用单一固定阈值，而采用“规则判据 $^ +$ 模型输出 $^ +$ 动态阈值”的综合方式。对于供电异常、堵转、严重过流等具有明确物理边界的故障，可优先使用电流、电压、温度等规则阈值快速识别；对于轻微异常、弱故障和趋势性退化，则结合随机森林分类概率、LSTM 预测风险值和健康评分进行判定；当多项指标同时越过预设边界时，报警等级自动上调。这样既保留了传统工业控制中规则判据的可解释性，也发挥了智能模型在复杂场景中的灵敏性。

# 7.2 预警后的控制响应逻辑

本项目的控制响应逻辑遵循“分级响应、先软后硬、保证安全”的原则，即根据报警等级和故障类型，逐步采取提示、限制、降载、停机等不同层次的措施，而不是对所有异常一律执行停机处理。这样既能避免误报导致的不必要停机，也更符合工业现场强调连续性和安全性的控制需求。已有工业设备健康监测系统研究表明，预警系统的输出应进一步服务于智能决策与远程协同模块，由系统根据动态阈值与故障趋势自动生成运维调整策略。

当系统处于“关注”级时，控制侧不直接改变设备运行方式，主要执行软响应策略，包括在 HMI 上提示异常趋势、记录事件日志、启动更高频率的数据采样与分析、向运维人员推送关注通知等。这一阶段的目标是尽量在不影响生产的前提下加强监测，以判断异常是否为偶然扰动还是持续性退化。若异常消失，系统可自动恢复为正常级；若异常持续积累，则提升为预警级。

当系统进入“预警”级时，应触发中等级控制响应。针对泵房场景，可根据故障类型采取不同策略：若判定为温升异常或轻微不平衡，可提示降低运行负荷、限制频繁启停并安排巡检；若判定为轴承或机械异常趋势，可提示准备备用泵或建议计划

性切换；若判定为电流波动异常但尚未达到高危水平，则可执行软限流、负载优化或预备切换逻辑。此时控制逻辑仍以“减缓故障发展、争取维修时间”为目标，不立即执行强制停机，但需要把算法输出结果传递给 PLC/EAE 侧，为可能的进一步联动作准备。

当系统进入“报警”级时，控制响应应优先保障设备与系统安全。对于严重过流、水泵堵转、绕组开路或连接机构断开等高风险故障，系统应触发声光报警，并根据预设逻辑执行联动动作，例如切断故障支路、停机保护、切换备用设备、禁止再次启动或发出人工确认请求。泵房设备故障诊断研究已表明，堵转、绕组开路、接触器开路等故障具有明确的电流电压特征，因此可作为高优先级联动对象。该类联动的核心目标是防止电机继续带病运行，避免故障扩大至配电或机械系统。

为了增强系统可用性，本文还设计了“人工优先、自动保护兜底”的控制策略。即在预警级阶段，系统以建议和提醒为主，允许人工确认是否采取调度或切换动作；在报警级阶段，若故障达到明确危险边界，则自动保护逻辑优先执行，同时保留人工复位或人工干预入口。这样的设计既符合工业控制对安全性的要求，也适应比赛后续阶段向 EAE/PLC 实际部署时的控制约束。

# 7.3 HMI 界面与状态显示方案

HMI 界面是本项目从“算法方案”走向“工业控制应用系统”的关键组成部分，其作用不仅是显示当前状态，更是把多源数据、模型结果和控制逻辑以工程人员易于理解的方式呈现出来。赛题明确要求 EAE 侧负责 HMI 显示，且最终系统需能展示从传感器输入、数据处理到控制执行和反馈的完整流程，因此 HMI 设计必须服务于状态监控、告警提示、趋势分析和人工交互。

本项目的 HMI 界面拟划分为四个核心区域。第一部分为“设备总览区”，显

示当前泵房设备总体状态，包括主泵/备用泵运行状态、系统级健康评分、当前报警等级以及通信状态。第二部分为“关键参数区”，实时显示振动、温度、电流等主要监测量及其变化趋势，并提供三相电流不平衡、温升速率等派生指标。第三部分为“智能诊断区”，显示当前故障类别、故障概率、趋势预测结果和预警建议，使运维人员不仅看到“报警了”，还能够理解“为什么报警”和“可能往哪个方向发展”。第四部分为“事件与操作区”，记录历史报警日志、模型判断结果、控制执行记录和人工干预按钮，为运维人员提供查询与确认功能。

在显示方式上，本文建议采用颜色分层和趋势图结合的方案。正常状态显示绿色，关注级显示蓝色或黄色提示，预警级显示橙色，报警级显示红色，并在界面中突出显示当前最主要的异常源。对于实时参数，采用数字 $^ +$ 仪表盘 $^ +$ 趋势曲线三种方式结合，以兼顾即时读取和趋势判断；对于模型输出，则可采用状态标签、风险概率条形图和健康评分折线图展示。若后续扩展到开放自动化实现，可在 HMI 中增加 MQTT通信状态、软件侧分析服务状态和 EAE 执行反馈状态，使整条链路可视化。

此外，HMI 界面还应支持基本的人机交互功能。例如，运维人员可查看某次报警的详细原因、查询某类故障的历史趋势、手动确认报警、启用或屏蔽某些联动动作，或查看系统给出的维护建议。已有多源实时健康监测系统中，数字孪生模块就承担了可视化展示和交互查询的功能，这说明将预测结果与交互界面结合，是工业预警系统发展的合理方向。

# 7.4 与 PLC 控制器的联动思路

为保证本项目具备从算法验证走向工业现场控制应用的能力，本文采用“软件分析侧 $^ +$ 控制执行侧”的联动思路。根据赛题要求，纯软件侧在 Debian 12 环境中独立开发算法模块、数据分析服务和 AI 推理引擎，不直接操作硬件，而是通过 MQTT 与

EAE 平台通信；EAE 侧负责现场 I/O 控制、PLC 程序和 HMI 显示。基于此，本项目将预警算法与 PLC/控制器的联动设计为松耦合式结构，即算法侧负责分析与决策输出，控制侧负责安全可靠的执行。

具体来说，软件分析侧接收来自传感器或边缘网关的多源数据，完成预处理、特征提取、故障分类与趋势预测后，输出包括设备编号、时间戳、故障类别、故障概率、健康评分、报警等级和建议动作在内的结构化结果。该结果通过 MQTT 以 JSON格式发布至约定主题，例如设备状态主题、预警结果主题和控制建议主题。赛题给出的开放自动化规范已明确推荐使用 MQTT、JSON 和 Mosquitto 作为通信方式和消息代理，因此本文采用同样路线，便于后续与 EAE 平台对接。

控制执行侧由 EAE/PLC 接收软件侧发布的预警结果后，根据预置逻辑完成动作判断。比如，当接收到“关注”级信息时，仅更新 HMI 状态并记录日志；当接收到“预警”级信息时，触发提示、限制启停或启动备用逻辑准备；当接收到“报警”级信息时，则执行声光报警、停机、切换或闭锁等动作。为了保证控制安全性，PLC 逻辑中仍保留传统保护条件与硬件联锁，不完全依赖 AI 模型输出。也就是说，智能算法负责“更早发现问题”，而 PLC/EAE 负责“更可靠执行保护”。这种分工方式可以降低纯数据驱动模型直接控制现场设备所带来的风险。

在工程实现层面，本项目建议将 PLC/EAE 侧联动变量分为三类：第一类为状态变量，如设备运行/停止、通信正常/异常、模型在线/离线；第二类为预警变量，如预警等级、故障代码、健康评分、趋势风险值；第三类为控制变量，如报警输出、允许启动标志、降载建议标志、切换备用设备命令等。通过这种标准化变量设计，可使算法模块与 PLC 程序之间的接口更加清晰，也便于后续增加更多故障类型或控制动作。

# 7.5 工业现场部署流程设计

为了使本项目在比赛后续阶段具备可展示、可调试、可扩展的工程实现路径，本文将工业现场部署流程划分为“硬件部署—通信接入—软件上线—控制联调—运行验证”五个阶段。这一部署思路既参考了多源工业健康监测系统中的边缘网关 、MQTT、数据融合和数字孪生结构，也严格对齐了赛题对开放自动化方向的实施边界。

第一阶段为硬件部署。在电机、轴承座和供电回路等关键位置安装振动、温度和电流传感器，并配置采集模块或边缘网关，实现多源数据实时获取。对于泵房实验平台，可优先部署少量关键测点以保证系统可运行；在后续扩展阶段，再逐步增加多点布置和更多辅助状态量。

第二阶段为通信接入。通过有线或无线方式将传感器数据接入边缘网关或工控机，在边缘侧完成初步去噪、格式统一和时间同步后，通过 MQTT 将数据转发至软件分析服务。已有多模态工业健康监测专利明确采用“多模态传感器—边缘计算网关—MQTT—实时流处理—特征工程—预测模型”的链路，这说明该部署方式具备较强工程可行性。

第三阶段为软件上线。在 Debian 12 环境中部署数据处理服务、特征工程服务和故障预警模型服务，配置 Mosquitto 代理及相关主题规则，使系统能够持续接收原始数据并输出预警结果。初赛阶段这些服务可先用 MATLAB/Python 离线验证；复赛和决赛阶段再逐步迁移到在线运行环境，形成接近实际工业现场的软件分析侧。

第四阶段为控制联调。将软件侧输出的预警结果与 EAE/PLC 程序进行联调，验证不同报警等级下的 HMI 显示、声光报警、运行限制和停机逻辑是否正确触发。该阶段重点不在于控制动作多复杂，而在于证明“算法输出能够被控制器可靠接收和执行”，即真正完成赛题要求的“决策输出 控制执行→HMI 反馈”闭环。

第五阶段为运行验证。在实验平台或样机系统上进行连续运行测试，记录正常

工况与典型异常工况下的系统表现，包括模型识别准确率、误报率、预警提前量、控制执行响应时间和 HMI 展示效果等。若运行中发现模型输出不稳定、通信延迟过高或联动逻辑不合理，可根据日志回溯优化算法参数、阈值设置和 PLC 程序，形成“部署—验证—修正”的迭代闭环。已有多源预警系统研究均强调，系统最终目标不仅是状态识别，还包括实时呈现、远程协同和运维策略优化，这与本项目的部署流程目标一致。

总体来看，本章提出的控制联动与工业应用设计，已经把前面的多源监测、特征处理和故障预警算法转化为面向工业现场的完整应用链条：上层有分级报警策略，中层有控制响应与 PLC/EAE 联动逻辑，下层有现场部署流程和 HMI 展示方案。这样设计的好处在于，本项目即使在初赛阶段主要以可研和仿真为主，整体方案也已经具备清晰的工程落地方向，并能够自然延展到复赛和决赛的样机实现与现场演示。

# 第八章 通信架构与软件系统设计

# 8.1 软件侧与控制侧分工

为满足赛题开放自动化应用的实现要求，同时保证故障预警算法与现场控制逻辑之间的职责清晰，本文采用“软件侧独立分析 $^ +$ 控制侧可靠执行”的双侧分工架构。根据赛题要求，软件侧运行于 Debian 12 环境，独立承担算法模块、数据分析服务和AI 模型推理引擎等功能，不直接操作现场硬件；控制侧则以 EAE 平台为核心，负责I/O 控制、PLC 程序和 HMI 显示，通过 MQTT 与软件侧交换数据。这样的分工方式既符合赛题边界，也能降低数据驱动模型直接控制现场设备所带来的安全风险。

在本文方案中，软件侧主要承担五类任务。第一类是数据接入与预处理，包括接收振动、温度、电流等多源数据，完成时间同步、格式统一、去噪、滤波、归一化与窗口分段。第二类是特征工程，包括提取时域、频域和时频域特征，并构建统一的多维特征向量。第三类是模型推理，包括 SVM 与随机森林的状态分类、LSTM 的趋势预测以及健康评分计算。第四类是预警判定与规则融合，即将模型输出进一步转换为故障类别、风险概率和分级告警结果。第五类是通信服务，即按照统一的主题结构和JSON 格式，将分析结果发布给控制侧，并接收控制侧返回的执行反馈与状态确认。现有多模态工业设备实时健康监测系统也采用了类似的思路：边缘侧与平台侧先完成数据清洗、特征工程和模型预测，再通过实时数据流支撑可视化与交互。

控制侧则主要负责现场级的可靠执行。其任务包括：接收软件侧发布的状态结果与预警信息；根据预设 PLC/EAE 逻辑完成报警、运行限制、停机保护或备用切换等动作；将执行状态反馈给 HMI 和软件侧；在软件侧异常、通信中断或模型不可用时，继续依靠本地保护逻辑保证设备安全。该分工符合工业控制“上层分析、下层兜底”

的基本原则，即智能算法负责“更早发现问题”，控制器负责“更可靠执行动作”。

从模块部署角度看，本文建议将系统划分为：传感接入模块、预处理模块、特征工程模块、推理模块、预警决策模块、MQTT 通信模块、PLC/EAE 控制模块和 HMI展示模块。其中，前六类模块主要部署于软件侧，后两类模块主要部署于控制侧。若后续需要增强系统规模，也可在 MQTT 之后加入缓存或流处理模块，但对比赛原型而言并非必需，保持结构简洁更有利于实现与展示。

# 8.2 MQTT 通信机制设计

赛题明确规定软件侧应用必须通过 MQTT 与 EAE 平台进行数据交换，并建议使用 Mosquitto 作为消息代理、JSON 作为数据格式，因此 MQTT 不仅是本项目的通信选项，更是与赛题技术路线一致的核心机制。MQTT 采用发布/订阅模式，能够实现数据提供方与数据使用方的解耦，非常适合多源监测、实时分析与控制反馈并存的工业场景。已有多模态工业健康监测系统也明确采用“边缘网关发布数据—MQTT 转发—平台侧融合分析”的方式，以兼顾通信轻量性、实时性和模块扩展性。

在本文设计中，MQTT 通信机制的核心思路是：传感接入或边缘预处理模块将原始或预处理后的多源数据发布到数据主题；软件分析服务订阅相应主题后完成特征提取、故障分类和趋势预测；分析完成后，再将结果发布到诊断结果主题、预警结果主题或控制建议主题；EAE/PLC 侧订阅这些主题并执行相应动作，同时将执行结果、设备状态和心跳信息返回到反馈主题；HMI 则通过订阅状态与报警主题实现界面刷新和日志记录。这样就形成了“数据上行—分析计算—结果下发—执行反馈”的闭环。

对于比赛实现而言，MQTT 机制的优势主要体现在三点。第一，通信链路清晰，便于向评委展示系统的模块化与工程化程度。第二，软件侧和控制侧天然解耦，便于后续独立调试模型和 PLC 逻辑。第三，系统扩展方便，后续若加入更多传感器、更多

设备编号或更多功能主题，只需按命名规范扩展，无需重构整体架构。因此，本文将MQTT 作为第八章通信设计的主线，而不采用强耦合的直连调用方式。

# 8.3 主题结构设计

赛 题 给 出 了 典 型 的 主 题 命 名 示 例 ， 如 eae/device/590d/status 和eae/app/vision/detection_result，说明主题设计应采用“层级化、可读化、语义清晰”的方式组织。基于这一要求，并结合本文故障预警系统的功能链路，本文建议采用“系统域 / 功能域 / 设备编号 / 数据类型”的四层主题结构，使原始数据、特征数据、诊断结果、控制命令和执行反馈彼此分离而又易于关联。

针对本项目，建议的主题结构如下：

```txt
eae/device/{device_id}/telemetry/raw  
eae/device/{device_id}/telemetry/feature  
eae/app/fault/{device_id}/diagnosis  
eae/app/fault/{device_id}/prediction  
eae/app/fault/{device_id}/alarm  
eae/control/{device_id}/command  
eae/control/{device_id}/ack  
eae/device/{device_id}/status  
eae/system/{device_id}/heartbeat 
```

上述主题结构的设计逻辑如下。telemetry/raw 用于上传原始或预处理后的多源监测数据，telemetry/feature 用于传输经过特征工程处理后的中间特征结果，diagnosis用于输出当前故障类别和分类概率，prediction 用于输出趋势预测结果和健康评分，alarm 用于发布分级报警信息，command 用于控制侧接收建议动作或执行请求，ack 用于控制侧返回动作确认，status 用于设备运行状态与在线状态同步，heartbeat 用于监测通信链路和服务存活状态。这样划分后，系统可以在不混淆不同语义数据的前提下完成全链路通信。该思路与赛题主题命名示例和现有多模态实时监测系统中的分层通信结构保持一致。

如果后续需要支持多设备或多泵机并行运行，只需通过 device_id 对不同设备进行区分，例如 pump01、pump02、motorA 等，即可在同一消息代理下实现多实例管理。若需要支持更细粒度的模块调试，还可以增加 debug、log 或 model 子主题，但在比赛方案中不宜过度复杂，保持“够用、清晰、可展示”更符合原型系统定位。

# 8.4 JSON 数据字段设计

赛题明确建议软件侧与 EAE 之间交换的数据采用 JSON 格式，因此，统一的数据字段定义是实现软件侧、控制侧与 HMI 高效联通的关键。已有多源智能预警系统通常会将设备状态、特征向量、预测结果、阈值判断和反馈状态组织为结构化字段，以便于实时解析、日志记录和后续追溯。

# （1）原始/预处理数据报文设计

原始或预处理数据报文主要用于承载设备侧上传的多源状态量，建议包含设备标识、时间戳、采样状态、多源信号值和数据质量标记等字段。推荐结构如下：

```json
{ "device_id": "pump01", "timestamp": "2026-03-30T10:15:20Z", "mode": "running", "signals": { "vibration_rms": 1.82, "temperature_motor": 64, "current_a": 12.6, "current_b": 12.9, "current_c": 14 }, "quality": { "sync.ok": true, "missing_rate": 0.00, "noise_flag": false } } 
```

这种结构可同时满足原始数据上传、预处理结果回传和后续日志留存需求，其

中 quality 字段用于表征时间同步、缺失率和噪声状态，便于模型侧决定是否直接推理或进入降级处理。现有多源预警方案中也强调了时间同步、深度清洗和异常过滤的重要性，因此本文将数据质量字段纳入设计。

# （2）诊断/预测结果报文设计

诊断与预测结果报文主要用于软件侧向控制侧和 HMI 发布算法结果。建议包含当前故障类别、分类概率、健康评分、趋势预测值、预警等级和建议动作等字段。推荐结构如下：

```jsonl
{ "device_id": "pump01", "timestamp": "2026-03-30T10:15:22Z", "diagnosis": { "fault_type": "winding_imbalance", "fault_prob": 0.87 }, "prediction": { "health_score": 0.42, "risk_prob": 0.78, "warning_level": "warning", "lead_time_s": 120 }, "action": { "suggested_action": "prepare.switch", "control_enable": true } 
```

该结构能够将“当前状态分类”和“未来趋势预测”同时表达出来，便于控制侧执行不同层次的逻辑，也便于 HMI 展示“当前发生了什么”和“接下来可能发生什么”。这与本文前文提出的“分类 $^ +$ 预测 $^ +$ 分级”算法输出相匹配。

# （3）控制反馈报文设计

控制反馈报文用于 EAE/PLC 侧向软件侧和 HMI 返回执行确认结果，建议包含动作类型、执行状态、反馈时间和附加说明。推荐结构如下：

{ 

```python
"device_id": "pump01",  
"timestamp": "2026-03-30T10:15:23Z",  
"command_id": "cmd_0001",  
"execution": {  
    "action": "alarm_on",  
    "status": "success",  
    "plc_mode": "auto"  
},  
"message": "Alarm output activated" 
```

通过引入 command_id，可实现诊断结果、控制命令和执行反馈的一一对应，便于系统日志追踪和异常排查。对于比赛演示而言，这种结构也更容易向评委展示“分析结果确实被控制器接收并执行”。

# 8.5 通信时序设计

通信时序设计的核心目标，是保证多源数据、算法结果和控制反馈在时间上形成清晰、可追踪、可闭环的交互流程。由于赛题明确强调完整链路展示，因此本项目需要的不只是“能发消息”，而是“消息在什么时刻、由谁发送、谁接收、如何反馈”都应有清晰定义。多源传感融合理论也指出，时间同步是多源系统可靠运行的基础，因此时序设计必须围绕时间一致性展开。

本文建议将通信时序分为五个阶段：

第一阶段为启动注册阶段。设备侧、软件分析侧和控制侧启动后，先向heartbeat 或 status 主题发送在线状态，完成模块自检和通信链路确认。若某一关键模块未上线，系统在 HMI 上标注“部分功能不可用”，但控制侧仍维持本地保护逻辑。

第二阶段为数据上行阶段。传感接入模块或边缘网关周期性向 telemetry/raw 主题 发 布 多 源 监 测 数 据 ； 软 件 侧 订 阅 该 主 题 ， 完 成 预 处 理 后 ， 可 按 需 要 向telemetry/feature 发布中间特征结果，用于调试、记录或下游服务复用。多模态工业健康监测系统中，边缘侧先进行清洗与初步处理，再通过 MQTT 向上游发布数据，本文

沿用这一思路。

第三阶段为分析推理阶段。软件侧在接收到足够窗口长度的数据后，完成故障分类与趋势预测，并将诊断结果发布到 diagnosis、prediction 和 alarm 等主题。若结果处于正常或关注级，可只更新状态与日志；若进入预警或报警级，则必须同步生成command 或控制建议消息，供控制侧处理。

第四阶段为控制执行阶段。EAE/PLC 侧订阅控制命令与预警结果主题，依据预设逻辑判断是否需要执行报警、限制、停机或切换动作。动作执行后，控制侧向 ack或 status 主题回传反馈结果，软件侧和 HMI 依据该反馈更新界面状态和日志。这样可以避免“算法已报警但控制未执行”的信息断裂。

第五阶段为持续监测与闭环优化阶段。系统在后续周期继续重复“采集—分析—执行—反馈”的流程，并把历史结果写入日志或数据库，用于模型复盘、阈值修正和运维分析。若后续加入数字孪生或知识更新模块，该阶段还可进一步演化为闭环优化系统。

可将上述时序概括为：

设备/边缘侧启动 $\rightarrow$ 发布心跳/状态

# 8.6 异常处理与容错机制

在工业控制应用中，通信架构的设计不能只考虑正常情况，还必须考虑数据异常、通信中断、模型失效和执行冲突等问题。已有多源融合与工业实时监测研究普遍强调：时间不同步、噪声污染、单模块故障、异常数据输入和通信链路不稳定，都会显著影响预警系统的可靠性，因此需要从数据层、通信层和控制层三个层面建立容错机制。

# （1）数据异常处理

当传感器数据存在缺失、突变、噪声污染或时间错位时，软件侧首先通过quality 字段进行标记，并进入预处理容错流程。对于短时缺失或单点异常，可采用插值、滑动平均或最近有效值替代；对于严重噪声或同步失败的数据片段，则跳过推理或降低置信度输出，防止“坏数据驱动错误告警”。多源传感融合理论和多模态工业监测方案都明确指出，时间同步和数据质量优化是融合系统可靠运行的基本前提。

# （2）通信异常处理

当 MQTT 消息代理异常、网络抖动或主题消息丢失时，系统需具备最基本的断连容错能力。本文建议：软件侧和控制侧均维持心跳检测机制，若连续若干周期未收到关键主题消息，则判定链路异常；边缘侧可对短时未成功发送的数据进行本地缓存并在链路恢复后补发；HMI 则在界面上明显提示“通信异常/结果滞后”。对于比赛原型而言，不需要实现过于复杂的分布式容灾，但至少要体现“断连可感知、短时可缓存、恢复可继续”的思想。相关工业监测系统中引入边缘网关与分层通信，正是为平衡实时性和可靠性。

# （3）模型异常处理

当软件侧模型服务异常退出、推理超时或输出结果明显失真时，系统不能因此失去全部保护能力。本文建议采用“模型失效，规则兜底”的机制：若模型服务不可用，则自动切换到基于阈值和规则的基础保护模式，继续依据电流超限、温度超限和严重不平衡等明确物理判据进行报警与保护。这样即使智能算法离线，系统仍具备最基本的安全防护能力。这种做法也符合前文“机理判据 $^ +$ 数据驱动模型”协同设计的原则。

# （4）执行冲突处理

在控制侧，若算法建议动作与现场硬件联锁、人工操作或 PLC 保护逻辑发生冲突，应以现场安全和本地保护优先。例如，若 AI 建议继续观察，但 PLC 已检测到硬件级过流，则仍应以 PLC 跳闸或停机保护为准；若运维人员手动切换到维护模式，则软件侧不再主动下发自动控制请求，只保留状态提示和日志记录。这样可以保证智能系统始终处于辅助决策和增强预警的位置，而不会破坏工业现场既有的安全控制体系。

# （5）系统容错总体思路

综上，本文建议本项目的容错机制遵循“数据可降级、通信可检测、模型可切换、控制有兜底”的原则。即：数据异常时可跳过或降置信推理，通信异常时可缓存和提示，模型异常时回退到规则保护，控制异常时以本地联锁和人工优先为准。通过这种分层容错设计，系统即使在非理想状态下，也能维持基本可用性和安全性，这对于比赛后续从初赛方案论证走向复赛、决赛实物实现尤其重要。

# 9.5 不同算法对比分析

为了验证本文算法设计的合理性，需要对不同模型在同一数据集上的表现进行横向比较。比较对象主要包括 SVM、随机森林和 LSTM，其中前两者重点比较分类能力，后者重点比较趋势预警能力，同时也可将其离散化后纳入分类指标比较。已有工业健康监测方案中，随机森林主要用于故障分类，LSTM 主要用于时间序列预测，两者在任务侧重点上本就存在差异，因此对比分析应从“分类精度 $^ +$ 趋势感知 $^ +$ 工程代价”三个维度展开，而不应只用单一准确率评价全部模型。

从分类性能角度看，SVM 适合做基线模型，通常在中小样本条件下能够取得较稳定结果，但对复杂非线性特征交互的刻画有限；随机森林对异构多源特征的适应性更强，通常在鲁棒性和可解释性上更具优势；LSTM 若直接用于多类分类，训练复杂度较高，对样本数量和时序完整性要求也更高。因而，若你们后续实测结果显示随机森林总体准确率和召回率高于 SVM，这是完全合理的。

从预警能力角度看，LSTM 的主要优势在于能利用连续时间窗信息预测健康趋势，因此其“预警提前量”通常优于纯静态分类模型。对于泵房设备这类渐进退化对象，LSTM 更适合回答“未来是否更危险”，而 SVM 和随机森林更适合回答“当前最像哪类故障”。因此，本节应明确：SVM/随机森林解决的是当前状态识别问题，LSTM 解决的是未来风险预测问题，三者并非简单替代关系，而是可形成互补。

从工程部署角度看，SVM 模型结构相对简单、推理速度较快；随机森林虽树数量较多，但在边缘侧或工控机上仍具备较好部署可行性；LSTM 由于涉及时间序列建模，训练时间更长、部署复杂度更高，但在趋势预警任务中更具价值。因此，本项

目推荐采用“SVM 做基线，随机森林做主分类，LSTM 做趋势预警”的组合方案，而不是单一依赖某一个模型。

# 9.7 存在问题与原因分析

尽管 MATLAB 仿真与模型训练能够初步验证本文方案的可行性，但从当前阶段来看，仍存在若干问题需要在后续复赛和决赛中进一步完善。

首先，当前仿真数据与真实工业现场数据之间仍存在差距。仿真模型能够较好地反映典型故障机理，尤其在电流、电压等电气变量上具有较强可控性，但对于复杂机械振动、温度传导滞后以及现场噪声扰动等因素的模拟仍不完全充分。因此，基于

仿真数据训练得到的模型虽能验证方法可行性，但其对实际泵房设备的泛化能力仍需通过实测数据进一步校准。泵房设备故障诊断论文虽然已用 MATLAB/Simulink 证明了多点电压电流信息融合方法的有效性，但也说明该阶段主要是仿真验证性质。

其次，多源数据同步与样本构建仍存在难度。振动、电流属于快变量，温度属于慢变量，不同采样频率和时间尺度下如何形成统一窗口，是多源建模中的关键问题。多传感器融合理论明确指出，时间同步和空间/对象对齐是信息融合的前提；若对齐不充分，即使后续模型复杂度再高，也会削弱预警效果。

再次，模型性能与工程约束之间存在权衡。随机森林在分类精度与鲁棒性上较有优势，LSTM 在趋势预测和预警提前量上更具价值，但 LSTM 的训练复杂度和部署成本也更高。若后续系统需部署到边缘侧或工控机上，还需进一步权衡模型复杂度、推理时延和现场可维护性。已有多模态工业健康监测方案采用分层架构，本质上也是为了解决“算法能力—实时性—工程可部署性”的平衡问题。

最后，当前阶段的预警分级与控制联动仍主要基于规则设计，尚未与完整的PLC/EAE 系统进行在线闭环验证。赛题后续阶段对开放自动化方向明确要求通过MQTT 与 EAE 平台交换数据，并能展示从数据处理到控制执行和 HMI 反馈的完整流程，因此下一阶段需要进一步完成通信联调与现场级闭环验证。

综上所述，本文 MATLAB 仿真与结果分析能够证明所提方案在故障识别和趋势预警方面具有较好的可行性，但仍需要在实测数据引入、多源同步优化、模型轻量化部署和 PLC/EAE 联动验证等方面继续完善。这些问题并不削弱方案的合理性，反而说明本项目具备从初赛仿真验证逐步走向复赛样机实现和决赛现场展示的明确技术演进路径。

# 10.1 本项目创新点

# （1）面向泵房场景的多源融合预警设计

现有泵房设备故障诊断研究已经证明，基于多点电压、电流信息融合，可以实现对供电电压超限、接触器开路、电机绕组开路、绕组不平衡与匝间短路、水泵堵转以及连接机构断开等典型故障的在线诊断。本文在此基础上进一步引入振动与温度监测维度，将原有偏电气量诊断的方法扩展为“电流/电压 $^ +$ 振动 $^ +$ 温度”的多源融合预警框架，使系统不仅能识别显性电气故障，还能增强对机械异常和热异常的感知能力。与单一电流或单一振动方案相比，该设计更符合复杂工业设备故障的多维表征特点。

# （2）“分类 $^ +$ 预测 $^ +$ 分级”的故障预警逻辑

赛题要求的是设备故障“预警系统”，而非单纯故障分类器。为此，本文没有停留在传统的静态识别上，而是构建了“随机森林/SVM 进行当前状态分类 $^ +$ LSTM 进行健康趋势预测 $^ +$ 动态阈值完成分级预警”的组合式算法框架。该框架能够同时回答两个问题：一是“当前最可能是什么故障”，二是“未来是否正在走向故障”。已有工业设备实时健康监测方案中，随机森林被用于状态分类，LSTM 被用于健康趋势预测，自编码器被用于异常检测，说明这种分类与预测结合的思路具有较好的理论支撑。本文将其进一步工程化为“正常—关注—预警—报警”的四级输出结构，使算法结果能够直接服务于控制联动与 HMI 展示。

# （3）面向开放自动化应用的双层系统架构

本项目从初赛阶段起就不是把方案写成一个孤立的 MATLAB 算法，而是按“软件分析侧 $^ +$ 控制执行侧”的双层架构来设计。软件侧在 Debian 12 环境中独立完成数据处理、特征工程、模型推理和预警服务，通过 MQTT/JSON 与 EAE 平台通信；控制侧由 EAE/PLC 完成 I/O 控制、逻辑执行与 HMI 显示。这种架构与赛题对开放自动化应用的技术要求高度一致，也使本项目具备从初赛可研、复赛样机到决赛现场演示的天然扩展能力。相较于只做算法、不考虑系统集成的参赛方案，这一点具有较明显的工程创新意义。

# （4）从“监测”走向“控制支撑”的闭环设计

很多故障诊断研究止步于“发现异常”，但本文进一步设计了报警等级划分、控制响应逻辑、PLC/EAE 联动和 HMI 显示方案，使预警结果能够转化为可执行的控制建议与运维决策。例如，轻微异常仅触发关注与趋势跟踪，预警状态触发检修建议和运行限制，严重报警则可执行停机保护或备用切换。这种从传感采集、模型预警到控制反馈的闭环设计，更符合赛题强调的“数据处理 决策输出 控制执行 ${ \bf \Gamma } \to \mathrm { H M I }$ 反馈”完整流程要求，也更接近真实工业控制应用场景。

综上，本项目的创新不在于片面追求某个模型结构新颖，而在于将泵房典型故障机理、多源感知、时序预警、开放自动化通信和控制联动整合为一套完整、可扩展的工业智能应用方案。

# 11.4 风险分析与应对措施

虽然本项目整体技术路线较清晰，但在实施过程中仍可能面临数据、算法、系统集成和时间管理等多方面风险。因此，有必要在前期就建立风险识别与应对机制，以提升项目推进的稳定性。

# （1）数据质量不足风险

由于初赛阶段主要依赖仿真数据，后续实测数据获取可能存在样本不足、标签不全、多源不同步等问题，从而影响模型训练效果和结果可信度。对此，本文的应对措施是：前期以 MATLAB/Simulink 仿真数据完成方法验证，形成稳定的算法主线；中后期逐步引入实验数据校正模型；对多源数据统一采用时间同步、重采样和质量标记机制；对样本不足类别通过窗口平移和轻量增强方式补充数据。

# （2）模型效果不稳定风险

不同模型在不同故障类型上的表现可能存在波动，尤其 LSTM 对样本规模和

参数设置较敏感，若训练数据不足可能导致趋势预测效果不理想。对此，本文采取“多模型组合、分层使用”的策略：SVM 作为基线模型，随机森林作为主分类模型，LSTM 主要负责趋势预测；若 LSTM 阶段性效果不理想，可先保留分类 $^ +$ 规则分级的方案，保证系统整体可用；同时通过交叉验证和参数调优提高模型稳定性。

# （3）通信与联调风险

在后续复赛或决赛阶段，软件侧、MQTT、中间件和 PLC/EAE 之间的接口对接可能出现消息延迟、字段不一致、订阅逻辑错误或执行反馈异常等问题。对此，应在前期就确定统一的主题结构、JSON 字段和通信时序，并先通过仿真消息和虚拟主题进行联调测试，再逐步接入真实控制器；同时引入心跳检测、消息确认和本地缓存机制，提高通信可靠性。赛题后续阶段对 MQTT 主题结构、数据字段和时序都有较高关注，因此这一风险需提前控制。

# （4）硬件集成与样机稳定性风险

样机搭建过程中，传感器布置不合理、采集链路不稳定、边缘节点性能不足等问题，都可能影响演示效果。对此，应优先保证“最小可用系统”先跑通，即先实现关键监测量采集和单设备预警，再逐步扩展更多功能；对硬件选型坚持“成熟、易用、可替代”的原则，避免过于复杂或不稳定的组件。

# （5）进度管理风险

比赛项目周期有限，而项目包含仿真建模、算法训练、系统通信、HMI 显示和报告撰写等多个环节，若缺乏清晰的进度控制，容易出现后期工作堆积。对此，应依照前述关键节点计划推进，每阶段设置负责人和验收标准；报告撰写与代码实现同步进行，避免全部工作压到最后；重要图表、模型结果和演示界面优先产出，确保任何时点都保有可展示成果。

# （6）答辩展示风险

即便技术方案成熟，若最终答辩表达不清、演示链路断裂，也会影响比赛效果。对此，应在项目推进后期专门安排展示演练，提前录制备份演示视频，准备“正常流程 $^ +$ 异常备用方案”，确保现场网络、软件或样机出现短时问题时仍可完成说明和展示。

总体来看，本项目的风险主要集中在“数据质量、模型稳定性、系统联调和时间管理”四个方面，但都可以通过“分阶段推进、统一接口、保留兜底方案、提前演练”进行有效控制。由于赛题本身具有清晰的阶段目标和技术边界，只要团队按计划推进，本项目具备较高的实施成功概率。

# 12.1 当前研究结论

本文围绕“工业控制智能应用”赛题中的“工控成套应用—设备故障预警系统”方向，结合工业泵房场景下水泵驱动电机及其配电回路的运行特点，提出了一种基于多源传感融合的设备故障预警系统总体方案。该方案以振动、温度、电流等监测量为主要输入，构建了“传感采集—数据处理—故障预警—控制联动—HMI 展示”的完整技术链路，并在总体架构、数据处理、算法设计、通信机制和控制联动等方面进行了系统论证。赛题明确指出，设备故障预警系统可基于振动、温度、电流等传感器数据，采用 LSTM、SVM、随机森林等模型进行异常预警，本文所提出的研究路线与赛题要求保持一致。

通过文献调研与方案分析可以得到以下结论。第一，泵房水泵驱动电机及配电系统属于典型工业控制对象，故障类型明确、工程需求突出，具备较强的应用价值和研究意义。已有泵房设备故障诊断研究已证明，多点电压电流信息融合能够对供电电压异常、绕组故障、水泵堵转和连接机构断开等故障进行有效识别，为本文的场景选取与故障定义提供了直接依据。

第二，多源传感融合是实现工业设备智能预警的重要技术基础。相较于单一传感器方案，振动、温度和电流等多源信息能够分别反映机械状态、热状态和电气状态，通过时间同步、特征提取和融合建模，可以更全面地刻画设备运行特征，提升故障识别的准确性和鲁棒性。多传感器融合理论以及多模态工业健康监测研究均表明，多源异构数据的互补性是提高复杂故障诊断和预警能力的关键。

第三，本文提出的“SVM 基线分类 $^ +$ 随机森林主分类 $^ +$ LSTM 趋势预测”的

算法组合具有较强合理性。其中，SVM 适合建立中小样本条件下的基线分类模型，随机森林适合处理多源异构特征并输出特征重要性，LSTM 则适合建模设备状态的时间演化过程，实现趋势预警和预警提前量分析。该算法框架不仅符合赛题给出的模型方向，也与现有工业设备健康监测研究中的常用方法高度一致。

第四，本文构建的系统方案不仅关注算法效果，还从一开始就考虑了工程化落地路径。在系统层面，方案采用“软件侧独立分析 $^ +$ 控制侧可靠执行”的双层架构，利用 MQTT 与 EAE 平台进行数据交换，并通过 HMI 实现状态显示和报警反馈。这使得本项目不仅可以作为一个 MATLAB 算法验证课题，也具备向开放自动化应用方向进一步扩展的能力。赛题已经明确要求软件侧通过 MQTT 与 EAE 平台交换数据，控制侧完成 PLC、I/O 与 HMI 功能，本文的设计与该要求完全契合。

综合来看，本文认为：基于多源传感融合的泵房设备故障预警方案，在研究对象选择、数据来源、算法设计、系统架构和工程扩展性方面均具备较好的合理性与可行性，能够作为本赛题初赛阶段的有效解决方案。

# 12.3 复赛与决赛进一步计划

在复赛阶段，项目将从“可研与仿真验证”进一步推进到“样机实现与联调验证”。重点工作包括：搭建实验型泵机或等效电机负载平台；完成振动、温度和电流传感器接入；采集在线多源运行数据，并与 MATLAB/Simulink 仿真数据联合构建训练

与测试样本；在软件侧实现预处理、特征提取、模型推理和预警分级功能；在控制侧实现基础控制逻辑与 HMI 状态显示；最终形成具有较高完成度的可演示样机系统。赛题对复赛明确要求形成包含完整理论设计框架和实际初步实验结果对比分析的初设报告，并提供完成度约 $7 0 \%$ 的实物作品，因此复赛阶段的核心目标是把“算法方案”变成“可运行系统”。

在决赛阶段，项目将进一步从“样机运行”推进到“完整工业控制智能应用系统”展示。重点工作包括：优化模型参数与阈值策略，提高准确率、召回率和预警提前量；完善软件侧与控制侧的 MQTT 通信链路，完成 JSON 字段、时序机制和异常处理机制联调；与 EAE/PLC 平台打通，实现从数据采集、故障预警到控制执行和 HMI反馈的完整闭环；补充经济性分析、市场前景和推广路径论证；整理完整源码 、README、配置说明和演示材料，形成可现场展示的最终系统。赛题对决赛明确要求可演示的完整实物系统、详细成果报告以及经济性和推广路径分析，本文方案的系统架构和通信设计正是为这一阶段预留接口和扩展空间。

从技术深化方向看，复赛与决赛阶段还可围绕以下几个方面继续完善。第一，引入更多实测数据，对仿真训练模型进行校正，提高模型对真实工况的泛化能力。第二，在保持随机森林 $^ +$ LSTM 主线不变的前提下，尝试引入自编码器、CNN-LSTM 等增强模型，提高早期弱故障识别能力。第三，进一步完善报警等级与控制动作之间的映射关系，增强预警结果对现场运维和控制决策的支撑能力。第四，在 HMI 和上位机层面增加趋势回放、日志分析和维护建议等功能，使系统更接近工程应用而非单纯比赛展示。已有多源工业健康监测研究和多源智能预警方案中，数字孪生、远程协同和持续学习机制已成为发展趋势，这些内容也可作为项目后续的潜在提升方向。

总体而言，本文项目后续的发展路径十分清晰：初赛阶段以可研报告和仿真验
